            <em>

            </em>

             bits

            <em>
             <a href="https://www.linuxmint.com/download.php" target="_blank">

            <em>
             Android
            </em>
           </p>

              USB
             </em>

             <em>
              Android
             </em>

            </li>

            <em>
             Linux
            </em>

            <em>
             live
            </em>

            <em>
             CD
            </em>

            <em>
             USB

            </em>

             WhatsApp-Key-DB-Extractor-master.zip
            </em>
            e dê um clique com o botão direito sobre o arquivo
            <em>
             WhatsAppKeyDBExtract.sh
            </em>
            e selecione
            <em>
             “Propriedades”,
            </em>
            marcando
            <em>
             “Permitir execução do arquivo como um programa”.
            </em>
           </p>
           <p>
            <img alt="" class="aligncenter size-full wp-image-4107" height="469" sizes="(max-width: 627px) 100vw, 627px" src="http://www.infocrime.com.br/infocrime/wp-content/uploads/2016/12/chmod.png" srcset="http://www.infocrime.com.br/infocrime/wp-content/uploads/2016/12/chmod.png 627w, http://www.infocrime.com.br/infocrime/wp-content/uploads/2016/12/chmod-300x224.png 300w" width="627"/>
           </p>
           <p>
            Na pasta onde descompactou  digite
            <em>
             “$
            </em>
            ./
            <em>
             WhatsAppKeyDBExtract.sh”
            </em>
            prosseguindo de acordo com as orientações do
            <em>
             script
            </em>
            (O
            <em>
             $
            </em>
            é o sinal da linha de comandos no
            <em>
             shell
            </em>
            ou
            <em>
             Terminal
            </em>
            ). Conecte o dispositivo quando solicitado. Os arquivos extraídos serão copiados na pasta
            <em>
             /Extracted
            </em>
            dentro da pasta onde o conteúdo foi extraído. Dentro dessa pasta você terá cinco arquivos, sendo o
            <em>
             msgstore.db
            </em>
            o arquivo com os diálogos,
            <em>
             wa.db
            </em>
            o arquivo com a agenda de contatos e o arquivo
            <em>
             whatsapp.cryptkey
            </em>
            a chave de criptografia para descriptografar os arquivos
            <em>
             msgstore-yyyy-mm-dd.x.db.crypt12
            </em>
            presentes na pasta
            <em>
             /sdcard
            </em>
            <em>
             /WhatsApp/Databases.
            </em>
           </p>
           <p>
            Pronto a extração foi realizada.
           </p>

           <a href="http://www.infocrime.com.br/2016/09/softwares-forenses-e-colisao-de-hash/">
            Softwares Forenses e colisão de hash
           </a>
          </h2>
          <div class="post-meta-top">
           <div class="alignleft">
            Postado por
            <span class="author">
             <a href="http://www.infocrime.com.br/author/marceloslcosta/" title="Marcelo Sampaio">
              Marcelo Sampaio
             </a>
            </span>
           </div>
           <div class="alignright">
            <div class="fb-like" data-font="lucida grande" data-href="http://www.infocrime.com.br/2016/09/softwares-forenses-e-colisao-de-hash/" data-layout="button_count" data-send="false" data-show-faces="true" data-width="150">
            </div>
            <div class="tw_share">
             <script src="//platform.twitter.com/widgets.js" type="text/javascript">
             </script>
             <a class="twitter-share-button" href="//twitter.com/share?url=http%3A%2F%2Fwww.infocrime.com.br%2F2016%2F09%2Fsoftwares-forenses-e-colisao-de-hash%2F&amp;via=Infocrimebr&amp;count=horizontal">
              Tweet
             </a>
            </div>
           </div>
          </div>
          <div class="entry">
           <p>
            Há poucos dias, lendo um parecer, me deparei com a seguinte afirmação:
           </p>
           <p style="padding-left: 30px;">
            <em>
             “… o algoritmo criptográfico MD5 já foi comprometido há vários anos, sendo que há mais de 10 anos esse algoritmo é sabidamente inseguro para o fim de criação de resumos criptográficos destinados à garantia de integridade de dados. Há mais de uma década não faz mais qualquer sentido usar hash MD5 para tentar garantir a integridade de cópias forenses de dados.”
            </em>
           </p>
           <p>
            Eu estou sempre aberto a discussões e a novos pontos de vista, afinal o aprendizado é um caminho, e compartilhar é o destino. Então não me surpreendo com tal afirmação, pois já fui abordado algumas vezes sobre colisões nos
            <em>
             hashes MD5
            </em>
            <em>
             (Message Digest 5)
            </em>
            e
            <em>
             SHA-1
            </em>
            <em>
             (Secure Hash Algorithm 1)
            </em>
            e a impossibilidade do suas utilizações na prática forense, entretanto não é bem assim.
           </p>
           <p>
            Os algoritmos de
            <em>
             hash
            </em>
            são utilizados em nossa atividade para fins de identificação, verificação e autenticação de arquivos de evidências, como imagens periciais, além da composição de
            <em>
             hashsets,
            </em>
            que são conjuntos de
            <em>
             hashes
            </em>
            como os da
            <a href="http://www.nsrl.nist.gov/">
             <em>
              NIST
             </em>
             <em>
              NSRL
             </em>
            </a>
            <em>
             (National Software reference Library)
            </em>
            utilizado pelo
            <em>
             KFF  Server
            </em>
            da
            <em>
             <a href="https://www.accessdata.com/">
              AccessData
             </a>
            </em>
            ,
            <em>
             Encase
            </em>
            da
            <em>
             <a href="https://www.guidancesoftware.com/">
              Guidance Software
             </a>
             ,
            </em>
            utilizados para separar arquivos conhecidos e
            <em>
            </em>
            assim diminuir o esforço de análise, indexação e pesquisa, processo esse conhecido popularmente como
            <em>
             <a href="http://www.infocrime.com.br/2011/01/o-que-e-denist/">
              deNIST
             </a>
            </em>
            . Outra utilização dos
            <em>
             hashsets
            </em>
            é na identificação de arquivos de imagens ligadas a prática de pedofilia e outros delitos.
           </p>
           <p>
            Em 2011, a
            <a href="https://tools.ietf.org/html/rfc6151">
             <em>
              RFC-6151
             </em>
            </a>
            passou a não recomendar o uso do
            <em>
             MD5
            </em>
            em tarefas onde a resistência a colisões fosse necessária.
           </p>
           <p class="newpage" style="text-align: justify; padding-left: 30px;">
            <em>
             “MD5 is no longer acceptable where collision resistance is required such as digital signatures. It is not urgent to stop using MD5 in other ways, such as HMAC-MD5; however, since MD5 must not be used for digital signatures, new protocol designs should not employ HMAC-MD5.” – RFC-6151
            </em>
           </p>
           <p class="newpage" style="text-align: justify;">
            O
            <em>
             MD5,
            </em>
            conforme recomendações, teve seu uso comprometido para fins criptográficos como armazenamento de
            <em>
             hashes
            </em>
            de senha e assinaturas digitais e as colisões do
            <em>
             SHA-1,
            </em>
            ainda estão em nível teórico, entretanto a utilização de ambos para fins de autenticação de arquivos ainda são válidas e utilizadas largamente nos
            <em>
             softwares
            </em>
            forenses conforme pode ser visto nas imagens abaixo.
           </p>
           <p style="text-align: center;">
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/Encase_6.png" rel="prettyPhoto[14]" title="Encase 6.19.7.2">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/Encase_6.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/Encase_Imager_7.png" rel="prettyPhoto[14]" title="Encase 7">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/Encase_Imager_7.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/FTK_Imager.png" rel="prettyPhoto[14]" title="FTK Imager 3.4.2.6">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/FTK_Imager.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/Guymager.png" rel="prettyPhoto[14]" title="Guymager 0.7.3-2">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/Guymager.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/OSForensics.png" rel="prettyPhoto[14]" title="OSForensics 3.2">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/OSForensics.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/Tableau_1.2.png" rel="prettyPhoto[14]" title="Tableau Imager 1.2">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/Tableau_1.2.png" width="50"/>
            </a>
            <a class=" prettyPhoto" href="http://www.infocrime.com.br/infocrime/images/fortools/Tableau_1.2_2.png" rel="prettyPhoto[14]" title="Tableau Imager 1.2">
             <img alt="" height="50" src="http://www.infocrime.com.br/infocrime/images/fortools/Tableau_1.2_2.png" width="50"/>
            </a>
           </p>
           <p class="newpage">
            Pesquisas realizadas em 2004, no sentido de comprovar tais colisões, obtiveram resultados positivos para arquivos de mesmo tamanho em
            <em>
             bytes
            </em>
            . Posteriormente, em outras pesquisas, essa condição foi quebrada, mas em compensação o tamanho final dos arquivos deveria permanecer o mesmo após o ataque. Cumprindo os requisitos determinados na pesquisa e passado pelo
            <em>
             MD5
            </em>
            o resultado obtido era o mesmo, entretanto ao passar o
            <em>
             SHA-1
            </em>
            , os resultados se mostravam diferentes. Ainda assim uma análise do arquivo permitiu identificar facilmente ao final do arquivo, o bloco de dados injetado, comparado ao arquivo original.
           </p>
           <p class="newpage" style="text-align: justify;">
            A probabilidade de colisão no
            <em>
             MD5
            </em>
            é de 1 em 2^128, imagine então, o esforço computacional necessário para efetuar a injeção de um arquivo em uma imagem forense e manter seu
            <em>
             hash MD5
            </em>
            inalterado?
           </p>
           <p class="newpage" style="text-align: justify;">
            Se fosse tão simples e comprometedor como argumentam os mais pessimistas, porque os desenvolvedores de
            <em>
             softwares
            </em>
            forenses continuam utilizando o
            <em>
             MD5
            </em>
            e o
            <em>
             SHA-1
            </em>
            para autenticar imagens forenses? Injetar um arquivo modificado em uma imagem forense e provocar uma colisão ao ponto de obter um mesmo valor de
            <em>
             hash
            </em>
            , mesmo que sendo o
            <em>
             MD5
            </em>
            , é uma tarefa que beira a ficção, pois geraria alterações dos
            <em>
             hashes
            </em>
            de pastas e subpastas que contivessem tal arquivo, assim como sua referência em outros arquivos de sistemas, de tal forma que obter o mesmo
            <em>
             hash
            </em>
            da aquisição não seria factível. Imagine, se você utilizar o
            <em>
             MD5
            </em>
            em conjunto com o
            <em>
             SHA-1
            </em>
            , o nível de segurança será infinitamente maior, pois a colisão a ser provocada após a injeção de dados forjados em uma imagem pericial, deveria manter os valores dos
            <em>
             hashes MD5
            </em>
            e
            <em>
             SHA-1
            </em>
            inalterados.
           </p>
           <p class="newpage" style="text-align: justify;">
            Em um exemplo muito utilizado para demonstrar isso, tomemos dois algoritmos: A soma e a multiplicação.
           </p>
           <pre class="brush: plain">1+1=2 e 1+2=3</pre>
           <p>
            A injeção de dados de forma conveniente podem provocar a colisão.
           </p>
           <pre class="brush: plain">1+1+5=7 e 1+2+4=7</pre>
           <p>
            Como resultado da colisão temos dois conjuntos de dados com mesmo resultado, mas alterando o algoritmo a ser aplicado aos dados, teremos outro resultado.
           </p>
           <pre class="brush: plain">1x1=1 e 1x2=2</pre>
           <p>
            Nos dados onde anteriormente obtivemos colisões, teremos um resultado bem diferente.
           </p>
           <pre class="brush: plain">1x1x5=5 e 1x2x4=8</pre>
           <p>
            Assim podemos ver quanto é complexo comprometer dois algoritmos simultaneamente, como o
            <em>
             MD5
            </em>
            e o
            <em>
             SHA-1.
            </em>
           </p>
           <p>
            Já proteger um banco de senhas com
            <em>
             hash MD5
            </em>
            é muito arriscado, pois podemos gerar uma lista de palavras em
            <em>
             hash MD5 (rainbow tables)
            </em>
            e promover um ataque de comparação de dicionários. Aqui a vulnerabilidade é grande, não porque o
            <em>
             MD5
            </em>
            seja ruim, mas sim porque ele é muito rápido, permitindo realizar um ataque em um curto espaço de tempo. Algoritmos de
            <em>
             hashes
            </em>
            bons para proteger senhas, devem ser lentos.
           </p>
           <p>
            Logo a utilização de
            <em>
             hashes
            </em>
            <em>
             MD5
            </em>
            e
            <em>
             SHA-1
            </em>
            , na minha humilde opinião e de outros tantos profissionais da área, continua a ser válida para a autenticação de arquivos de imagens forenses.
           </p>
           <p>
            É possível encontrar grande quantidades de artigos sobre o tema pela internet, mas apenas para ajudar, vejam os artigos a seguir:
           </p>
           <ul>
            <li>
             <a href="http://www.forensicmag.com/article/2008/12/hash-algorithm-dilemma%E2%80%93hash-value-collisions">
              http://www.forensicmag.com/article/2008/12/hash-algorithm-dilemma-hash-value-collisions
             </a>
            </li>
            <li>
             <a href="https://digital-forensics.sans.org/blog/2009/01/07/law-is-not-a-science-admissibility-of-computer-evidence-and-md5-hashes/">
              https://digital-forensics.sans.org/blog/2009/01/07/law-is-not-a-science-admissibility-of-computer-evidence-and-md5-hashes/
             </a>
            </li>
            <li>
             <a href="http://federalevidence.com/pdf/2013/02Feb/EE-4thAmSearch-Power%20of%20Hash.pdf">
              http://federalevidence.com/pdf/2013/02Feb/EE-4thAmSearch-Power of Hash.pdf
             </a>
            </li>
           </ul>
           <p>
            Até o próximo.
           </p>

           <a href="http://www.infocrime.com.br/2016/08/analisando-o-log-de-eventos-parte-ii/">
            Analisando o Log de Eventos – Parte II
           </a>
          </h2>
          <div class="post-meta-top">
           <div class="alignleft">
            Postado por
            <span class="author">
             <a href="http://www.infocrime.com.br/author/marceloslcosta/" title="Marcelo Sampaio">
              Marcelo Sampaio
             </a>
            </span>
           </div>
           <div class="alignright">
            <div class="fb-like" data-font="lucida grande" data-href="http://www.infocrime.com.br/2016/08/analisando-o-log-de-eventos-parte-ii/" data-layout="button_count" data-send="false" data-show-faces="true" data-width="150">

            <div class="tw_share">
             <script src="//platform.twitter.com/widgets.js" type="text/javascript">
             </script>
             <a class="twitter-share-button" href="//twitter.com/share?url=http%3A%2F%2Fwww.infocrime.com.br%2F2016%2F08%2Fanalisando-o-log-de-eventos-parte-ii%2F&amp;via=Infocrimebr&amp;count=horizontal">
              Tweet
             </a>
            </div>
           </div>
          </div>
          <div class="entry">
           <p>
            Recentemente me deparei com um questionamento a um trabalho realizado há cerca de três anos, relacionado a alguns registros armazenados no arquivo
            <em>
             System.evtx
            </em>
            do
            <em>
             Log
            </em>
            de Eventos do
            <em>
             Microsoft Windows 7
            </em>
            . Um dos registros dizia o seguinte:
           </p>
           <pre class="brush: plain">06/02/2013 12:07:25 – A hora do sistema mudou de 2013-02-06T15:07:25.500000000Z para 2013-02-06T01:51:27.510396000Z.
05/02/2013 22:51:22 – O sistema está entrando em suspensão. Motivo da Suspensão: Botão ou Tampa</pre>
           <p>
            Sabendo que a data e hora do registro está no padrão de horário local e as datas e horas alteradas estão no padrão
            <em>
             UTC (Universal Time Cordinated)
            </em>
            teríamos então, em primeira análise, que houve uma atualização de um tempo mais recente para um tempo mais antigo, ou seja, que o relógio foi atrasado para um horário anterior, não deixando claro o fato ocorrido, visto que o sistema não executaria tal ação sozinho. Observando o registro nos seus detalhes observa-se que, o sistema registra a mudança de horário no seguinte formato:
           </p>
           <pre class="brush: plain">TimeCreated [SystemTime] – A hora do sistema mudou de EventData [NewTime] para EventData [OldTime]</pre>
           <p>
            Seria coerente se fosse ao contrário. Primeiro
            <em>
             OldTime
            </em>
            e depois
            <em>
             NewTime. S
            </em>
            abendo que não houve manipulações do sistema, resolvi realizar uma análise mais profunda sobre o caso, até porque o objetivo inicial da perícia era outro e a citação do
            <em>
             log
            </em>
            de eventos tinha como único motivo demonstrar que o usuário tinha por hábito suspender o uso do computador e não desligá-lo.
           </p>
           <p>
            Inicialmente realizei uma pesquisa em diversos
            <em>
             sites
            </em>
            relacionados ao suporte do
            <em>
             Microsoft Windows
            </em>
            e apenas em um fórum brasileiro, alguém fazia menção ao mesmo problema, imputando o fato a um usuário que supostamente obteve acesso ao servidor e realizou tal alteração. Em inglês, não havia uma linha sequer sobre o assunto, o que me deixou desconfiado do problema. Seria possível que o sistema teria sido manipulado pelo usuário e produzido tal resultado? Foquei no fórum brasileiro e li todas as dúvidas e sugestões acerca do fato e nada. Eles não haviam chegado a nenhuma conclusão.
           </p>
           <p>
            A composição do registro do evento era estranha. Como apontar uma mudança de horário indicando primeiramente a nova hora e depois a hora antiga? Será que os campos
            <em>
             NewTime
            </em>
            e
            <em>
             OldTime
            </em>
            estavam invertidos? Qual seria o problema então? Aí você tem que ser “o perito”, essa é sua função. Antes de atribuir o problema a algo que você desconhece ou ignora, pense, analise e refaça o trabalho que está deixando tanta dúvida.
           </p>
           <p>
            Inicialmente conferi, como disse no
            <em>
             post
            </em>
            anterior, o campo
            <em>
             EventRecordID,
            </em>
            até porque a mudança de horário alteraria o posicionamento dos registros quando classificados pelo momento da gravação do registro. Nada errado. Tudo seguia a sequência normal dos eventos, que nesse caso possuía a numeração 172401, 172402, etc.
           </p>
           <p>
            Quando o registro é gravado no disco, possui pouco conteúdo em formato legível para humanos
            <em>
             (human readable),
            </em>
            sendo que sua composição é realizada pelos visualizadores, através da combinação de modelos pré-definidos armazenados em
            <em>
             DLLs
            </em>
            e
            <em>
             EXEs,
            </em>
            com os dados existentes nos arquivos
            <em>
             EVT
            </em>
            e
            <em>
             EVTX
            </em>
            . Isso é percebido claramente quando copiamos os arquivos
            <em>
             EVT/EVTX
            </em>
            para analisar em outro computador que não possua os mesmos
            <em>
             softwares
            </em>
            do computador alvo da análise, gerando mensagens do tipo:
           </p>
           <p style="padding-left: 30px;">
            “
            <em>
             The description for Event ID ( xxxx ) in Source ( yyyyyy ) could not be found.
            </em>
            <br/>
            <em>
             Either the component that raises this event is not installed on the computer or the installation is corrupted. You can install or repair the component or try to change Description Server.
            </em>
            “
           </p>
           <p>
            Como utilizei um equipamento dotado do
            <em>
             Microsoft Windows 7
            </em>
            em português, e vi que nenhum
            <em>
             site
            </em>
            de língua inglesa fazia menção ao problema, resolvi realizar a análise detalhada da composição, trocando o sistema operacional por um
            <em>
             Microsoft Windows 7
            </em>
            com idioma em inglês, afinal este tipo de evento é gerado pelo próprio sistema operacional e não por uma aplicação de terceiros instalada nele. Feito isso abri o arquivo
            <em>
             System.evtx
            </em>
            no
            <em>
             snap-in
            </em>
            do
            <em>
             Event Viewer (
            </em>
            visualizador de eventos) e busquei os mesmos registros, obtendo o seguinte resultado:
           </p>
           <pre class="brush: plain">06/02/2013 12:07:25 – The system time has changed to 2013-02-06T15:07:25.500000000Z from 2013-02-06T01:51:27.510396000Z.
05/02/2013 22:51:22 – The system is entering in sleep. Sleep Reason: Button or Lid</pre>
           <p>
            Comparei então os detalhes e o formato do evento onde vi que a mensagem era composta assim:
           </p>
           <pre class="brush: plain">TimeCreated [SystemTime] – The system time has changed to EventData [NewTime] from EventData [OldTime]</pre>
           <p>
            Um erro na tradução do pacote de idioma produzia um registro incoerente. Na tradução
            <em>
             “to”
            </em>
            transformou-se em
            <em>
             “de”
            </em>
            e
            <em>
             “from”
            </em>
            em
            <em>
             “para”.
            </em>
           </p>
           <div class="wp-caption aligncenter" id="attachment_3974" style="width: 655px">
            <img alt="Comparação da descrição" class="wp-image-3974 size-full" height="212" src="http://www.infocrime.com.br/infocrime/images/evt/EVT_COMP.png" width="645"/>
            <p class="wp-caption-text">
             <em>
              Descrição do evento em português e inglês e suas diferenças na tradução
             </em>

          <h2>
           <a href="http://www.infocrime.com.br/2016/08/whatsapp-com-crypt12-android/">
            WhatsApp com crypt12 – Android
           </a>
          </h2>
          <div class="post-meta-top">
           <div class="alignleft">
            Postado por
            <span class="author">
             <a href="http://www.infocrime.com.br/author/marceloslcosta/" title="Marcelo Sampaio">
              Marcelo Sampaio
             </a>
            </span>
           </div>
           <div class="alignright">
            <div class="fb-like" data-font="lucida grande" data-href="http://www.infocrime.com.br/2016/08/whatsapp-com-crypt12-android/" data-layout="button_count" data-send="false" data-show-faces="true" data-width="150">
            </div>
            <div class="tw_share">
             <script src="//platform.twitter.com/widgets.js" type="text/javascript">
             </script>
             <a class="twitter-share-button" href="//twitter.com/share?url=http%3A%2F%2Fwww.infocrime.com.br%2F2016%2F08%2Fwhatsapp-com-crypt12-android%2F&amp;via=Infocrimebr&amp;count=horizontal">
              Tweet
             </a>
            </div>
           </div>
          </div>
          <div class="entry">
           <p>
            Passou mais de um ano desde o último post sobre o
            <em>
             WhatsApp
            </em>
            . Apareceu a criptografia ponta a ponta assegurando que somente os interlocutores possam ler as mensagens e ter acesso as mídias transmitidas, impedindo a interceptação, os arquivos
            <em>
             Crypt
            </em>
            já estão na versão 12, e outras novidades mais.
           </p>
           <p>
            Mas continuamos a abordar como extrair os arquivos necessários presentes no dispositivo para se obter acesso às mensagens e a chave de criptografia necessária a abrir os arquivos
            <em>
             msgstore-yyyy-mm-dd.x.db*
            </em>
            .
           </p>
           <p>
            Conforme vimos no
            <a href="http://www.infocrime.com.br/2015/01/whatsapp-com-crypt7-e-crypt8-android/" rel="noopener" target="_blank">
             <em>
              post
             </em>
             anterior
            </a>
            sobre o
            <em>
             WhatsApp
            </em>
            , os desenvolvedores do aplicativo desabilitaram a permissão de
            <em>
             backup
            </em>
            pelo
            <em>
             ADB,
            </em>
            de seus arquivos, no
            <em>
             app manifest,
            </em>
            exigindo a substituição do
            <em>
             app
            </em>
            pela versão sem a restrição de permissão de
            <em>
             backup.
            </em>
            Mais uma vez temos novidades com o lançamento de mais uma solução para extração da chave de criptografia. Trata-se de um
            <em>
             script
            </em>
            que realiza os passos do
            <em>
             post
            </em>
            anterior, mas de forma mais automatizada.
           </p>
           <p>
            REQUISITOS:
           </p>
           <ul>
            <li>
             Sistema
             <em>
              Windows
             </em>
             Vista, 7, 8, 10, Mac OS X, Linux;
            </li>
            <li>
             <em>
              <a href="http://www.java.com/en/download/manual.jsp" rel="noopener" target="_blank">
               Java
              </a>
              ;
             </em>
            </li>
            <li>
             <em>
              <a href="http://forum.xda-developers.com/showthread.php?t=2588979" rel="noopener" target="_blank">
               ADB
              </a>
              (Android Debug Bridge);
             </em>
            </li>
            <li>
             Ative a “Depuração
             <em>
              USB
             </em>
             ” em “Opções do desenvolvedor”. Caso não apareça no menu, clique seguidamente na opção “Sobre o telefone\Número da versão” até efetuar o desbloqueio da “Opções do desenvolvedor”;
            </li>
            <li>
             <em>
              Android
             </em>
             com versão 4.0 ou superior
            </li>
           </ul>
           <p>
            Preenchido os requisitos baixe os arquivos necessários
            <a href="http://click.xda-developers.com/api/click?format=go&amp;jsonp=vglnk_147152826974013&amp;key=f0a7f91912ae2b52e0700f73990eb321&amp;libId=is0dx8mo01000n4o000DA9zfh0j3m&amp;loc=http%3A%2F%2Fforum.xda-developers.com%2Fshowthread.php%3Ft%3D2770982&amp;v=1&amp;out=https%3A%2F%2Fgithub.com%2FEliteAndroidApps%2FWhatsApp-Key-DB-Extractor%2Farchive%2Fmaster.zip&amp;ref=http%3A%2F%2Fforum.xda-developers.com%2Fsitesearch.php%3Fq%3Dwhatsapp%2520extractor&amp;title=%5BTool%5D%20WhatsApp%20Key%2FDB%20Extractor%20%7C%20CRYPT6-12%E2%80%A6%20%7C%20Android%20Development%20and%20Hacking&amp;txt=WhatsApp-Key-DB-Extractor-master.zip" rel="noopener" target="_blank">
             aqui
            </a>
            . Descompacte e execute o
            <em>
             arquivo WhatsAppKeyExtract.bat
            </em>
            , conectando o dispositivo quando solicitado. Os arquivos extraídos serão copiados na pasta
            <em>
             /Extracted
            </em>
            dentro da pasta onde o conteúdo foi extraído. Dentro da pasta você terá cinco arquivos, sendo o
            <em>
             msgstore.db
            </em>
            o arquivo com os diálogos,
            <em>
             wa.db
            </em>
            o arquivo com a agenda de contatos e o arquivo
            <em>
             whatsapp.cryptkey
            </em>
            a chave de criptografia para descriptografar os arquivos
            <em>
             msgstore-yyyy-mm-dd.x.db.crypt12
            </em>
            presentes na pasta
            <em>
             /sdcard
            </em>
            <em>
             /WhatsApp/Databases.
            </em>
           </p>
           <p>
            Eu utilizo preferencialmente o
            <em>
             Linux
            </em>
            como ambiente para realizar as extrações. No
            <em>
             Windows
            </em>
            encontro muitos problemas e até deixar o ambiente 100%, tenho um trabalho considerável.
           </p>
           <p>
            Só para lembrar: Não é possível abrir os arquivos criptografados sem a chave utilizada para a codificação.
           </p>
           <p>
            Até o próximo.
           </p>

String.prototype.trim = function() {
  return this.replace(/^\s+|\s+$/g,"");
}

function createCookie(name,value,days) {
  if (days) {
    var date = new Date();
    date.setTime(date.getTime()+(days*24*60*60*1000));
    var expires = "; expires="+date.toGMTString();
  } else {
    var expires = "";
  }
  document.cookie = name+"="+value+expires+"; path=/";
}

function readCookie(name) {
  var nameEQ = name + "=";
  var ca = document.cookie.split(';');
  for(var i=0;i < ca.length;i++) {
    var c = ca[i];
    while (c.charAt(0)==' ') {
      c = c.substring(1,c.length);
    }
    if (c.indexOf(nameEQ) == 0) {
      return c.substring(nameEQ.length,c.length);
    }
  }
  return null;
}

function eraseCookie(name) {
  createCookie(name,"",-1);
}

function addExpandCollapse(id, expandSym, collapseSym, accordion) {
  jQuery('#' + id + ' span.expand').on('click.collapsingArchives', function() {
    if (accordion==1) {
      var theDiv = jQuery(this).parent().parent().find('span.collapse').parent().children('div');
      jQuery(theDiv).hide('normal');
      jQuery(this).parent().parent().find('span.collapse').removeClass('collapse').addClass('expand');
      createCookie(theDiv.attr('id'), 0, 7);
      jQuery(this).parent().parent().find('span.sym').html(expandSym);
      //jQuery('#' + id + ' .expand .sym').html(expandSym);
    }
    expandArch(this, expandSym, collapseSym);
    return false;
  });
  jQuery('#' + id + ' span.collapse').on('click.collapsingArchives', function() {
    collapseArch(this, expandSym, collapseSym);
    return false;
  });
}

function expandArch(symbol, expandSym, collapseSym) {
    var theDiv = jQuery(symbol).parent().children('div');
    try {
      if (collapsItems[jQuery(theDiv).attr('id')]!='') {
        jQuery(theDiv).html(collapsItems[jQuery(theDiv).attr('id')]).show('normal');
      } else {
        jQuery(theDiv).show('normal');
      }
    } catch (err) {
      jQuery(theDiv).show('normal');
    }
    jQuery(symbol).removeClass('expand').addClass('collapse');
    jQuery(symbol).children('.sym').html(collapseSym);
    jQuery(symbol).off('click.collapsingArchives').on('click.collapsingArchives', function() {
      collapseArch(symbol, expandSym, collapseSym);
      return false;
    });
  createCookie(theDiv.attr('id'), 1, 7);
}
function collapseArch(symbol, expandSym, collapseSym) {
    var theDiv = jQuery(symbol).parent().children('div');
    jQuery(theDiv).hide('normal');
    jQuery(symbol).removeClass('collapse').addClass('expand');
    jQuery(symbol).children('.sym').html(expandSym);
    jQuery(symbol).off('click.collapsingArchives').on('click.collapsingArchives', function() {
      expandArch(symbol, expandSym, collapseSym);
      return false;
    });
  createCookie(theDiv.attr('id'), 0, 7);
}
