          Android Studio 3.2 Beta
         </a>
        </h2>
        <div class="post-header">

<em>Posted by <a href="https://www.google.com/+JamalEason">Jamal Eason</a>, Product Manager, Android</em>

<p>
<a href="https://2.bp.blogspot.com/-lieVqC3AvxA/Wywnajrm7_I/AAAAAAAAFhs/xAsnax51askGLk1o0IiHHEpByD1yqYkaQCLcBGAs/s1600/pasted%2Bimage%2B0%2B%25287%2529image1.png" imageanchor="1" ><img border="0" data-original-height="225" data-original-width="250" id="floatRight" src="https://2.bp.blogspot.com/-lieVqC3AvxA/Wywnajrm7_I/AAAAAAAAFhs/xAsnax51askGLk1o0IiHHEpByD1yqYkaQCLcBGAs/s1600/pasted%2Bimage%2B0%2B%25287%2529image1.png" /></a></p>
<p>
Starting today, you can <a href="https://developer.android.com/studio/preview/">download</a> Android Studio 3.2 Beta. Previewed at Google I/O 2018, the latest release of the official Android IDE is focused on helping onboard you to all the new features launched around Google I/O -- <a href="http://developer.android.com/jetpack">Android JetPack</a>, <a href="https://developer.android.com/preview/">Android P Developer Preview</a>, and the new <a href="http://d.android.com/appbundle">Android App Bundle</a> format. There are also several other exciting new features included in Android Studio 3.2 to accelerate your app development, such as Emulator Snapshots and the Energy Profiler.
</p>
<p>
As the usage of Android Studio has grown in the 3.5 years since version 1.0, we have also become increasingly obsessed with quality. We continue to invest in quality because we know that millions of app developers spend almost everyday in Android Studio and need a reliable set of tools.  Stability, build times, and other quality work will be the primary focus for our next release once we finish Android Studio 3.2. We also did not want to wait, so we have made checkins to address <a href="https://twitter.com/androidstudio/status/1004800015481561088">memory leaks</a> and performance issues as well as fixed more than 450 bugs. Thank you for the continued feedback and please keep it coming so we can focus on the areas you care about most in the next version of Android Studio. If want to try out the latest features, and assess the improvements in quality, you can <a href="https://developer.android.com/studio/preview/">download</a> Android Studio on the beta release channel. 
</p>
<p>
<strong><span style="text-decoration:underline;">What is inside of Android Studio 3.2</span></strong>
</p>
<p>
Building on the <a href="https://android-developers.googleblog.com/2018/05/android-studio-3-2-canary.html">canary release</a> of Android Studio 3.2, the Beta release includes:
</p><ul>

<li><strong>Android App Bundle support </strong>- The <a href="http://d.android.com/appbundle">Android App Bundle</a> is a new publishing format that uses the Google Play's Dynamic Delivery, which delivers a smaller, optimized APK that only contains the resources needed for a specific device. Without any code changes, you can take advantage of the app size savings of an Android App Bundle by navigating to <strong>Build </strong>&#8594; <strong>Build Bundle / APK</strong>  or <strong>Build</strong> &#8594; <strong>Generate Signed Bundle / APK. </strong>  </li></ul>

<p>
<a href="https://3.bp.blogspot.com/-UXJn2NlJS5M/WywnrxQc5oI/AAAAAAAAFh0/RvpeN3jHvCMOy2SOE3IFGTr_Q_lHiNC0wCLcBGAs/s1600/unnamed%2B%25288%2529image2.png" imageanchor="1" ><img border="0" data-original-height="648" data-original-width="854" id="imgFull" src="https://3.bp.blogspot.com/-UXJn2NlJS5M/WywnrxQc5oI/AAAAAAAAFh0/RvpeN3jHvCMOy2SOE3IFGTr_Q_lHiNC0wCLcBGAs/s1600/unnamed%2B%25288%2529image2.png" /></a></p>
<p id="imgCaption">
<em>Build Android App Bundle</em>
</p><ul>

<li><strong>Emulator Snapshots -  </strong>With Android Studio 3.2 you can create snapshots at any emulator state and then start a snapshot in under 2 seconds. You can pre-configure an Android Virtual Device (AVD) snapshot with the apps, data and settings that you want and then repeatedly go back to the same snapshot. <a href="https://developer.android.com/studio/preview/features/#emulator_improvements">Learn more</a>. </li></ul>

<p>
<a href="https://3.bp.blogspot.com/-wAjOv0sDIWg/Wywn7hoqxuI/AAAAAAAAFh4/Y0IcoaE_AhYNeNsO4OPHz9QaKmYhVs3ywCLcBGAs/s1600/unnamedimage3.gif" imageanchor="1" ><img border="0" data-original-height="284" data-original-width="512" id="imgFull" src="https://3.bp.blogspot.com/-wAjOv0sDIWg/Wywn7hoqxuI/AAAAAAAAFh4/Y0IcoaE_AhYNeNsO4OPHz9QaKmYhVs3ywCLcBGAs/s1600/unnamedimage3.gif" /></a></p>
<p id="imgCaption">
<em>Android Emulator Snapshots</em>
</p><ul>

<li><strong>Energy Profiler </strong>-  The new <a href="https://developer.android.com/studio/profile/energy-profiler.html">Energy Profiler</a> in the performance profiler suite can help you understand the energy impact of your app on an Android device. You can now visualize the estimated energy usage of system components, plus inspect background events that may contribute to battery drain.  </li></ul>

<p>
<a href="https://2.bp.blogspot.com/-FmzRIqmlRXA/WywoLoih5PI/AAAAAAAAFiA/OUnBYHVWHAgkkOBwTD2qIGYrD8TMbW77ACLcBGAs/s1600/energy_profilerimage4.gif" imageanchor="1" ><img border="0" data-original-height="744" data-original-width="1270" id="imgFull" src="https://2.bp.blogspot.com/-FmzRIqmlRXA/WywoLoih5PI/AAAAAAAAFiA/OUnBYHVWHAgkkOBwTD2qIGYrD8TMbW77ACLcBGAs/s1600/energy_profilerimage4.gif" /></a></p>
<p id="imgCaption">
<em>Energy Profiler</em>
</p>

Check out the full write-up of all the major features organized by development flow listed below and on the <a href="https://android-developers.googleblog.com/2018/05/android-studio-3-2-canary.html">canary blog</a>:
</p>

<p>
<strong><span style="text-decoration:underline;">Sessions at Google I/O '18</span></strong>
</p>
<p>
With the release of Android Studio 3.2 at Google I/O '18, the Android Studio team also presented a series of sessions about Android Studio. Watch the following videos to see the latest features in action and to get tips & tricks on how to use Android Studio:
</p><ul>

<li><a href="https://youtu.be/WxAZk7A7OkM">What's new in Android development tools</a>
<p>

    An overview of all the recent features in Android Studio for Android app developers. The session includes demos and a tour de force presentation of relevant features that will accelerate developers' workflow on the latest Android APIs.
</p>

<li><a href="https://youtu.be/N5xONyp69eU">What's new with the Android build system</a>
<p>

    A deep dive into the new features of the Android build system.
</p>

<li><a href="https://youtu.be/ytZteMo4ETk">What's new with ConstraintLayout and Android Studio design tools</a>
<p>

    This session covers the new features in ConstraintLayout 2.0, as well as the latest functionality added in Android Studio design tools, highlighting how to effectively take advantage of them for designing, prototyping, and building a graphical user interface application.
</p>

<li><a href="https://youtu.be/O5V9ZSL0BsM">Improve app performance with Android Studio Profilers</a>
<p>

    This talk demonstrates how to diagnose and troubleshoot performance problems with your app using Android Studio Profilers. It covers examples of how to use the CPU, memory, network profilers, and highlight what's new.
</p>

<li><a href="https://youtu.be/gGOOkk2y_Ss">Best practices using compilers in Android Studio</a>
<p>

    This session discusses in-depth what is happening to the various compilers used in Android: Java 8 language feature desugaring, the new dexer (D8) and shrinker (R8), and work done on the Kotlin compiler for Android use.
</p>

<li><a href="https://youtu.be/8GCXtCjtg40">Android Jetpack: manage UI navigation with Navigation Controller</a>
<p>

    This session discusses how to use the Navigation Editor in Android Studio, XML or the Java API to define your navigation graph, and how that simplifies navigating around your app and handling deep linking.
</p></ul>

<p>
<strong><span style="text-decoration:underline;">Download & Feedback </span></strong>
</p>
<p>
Download the latest version of Android Studio 3.2 from the beta channel <a href="https://developer.android.com/studio/preview/">download page</a>. If you are using a previous versions of Android Studio, make sure you update to Android Studio Beta 1 or higher.  If you also want to maintain a stable version of Android Studio, you can run the stable release version and beta release versions of Android Studio at the same time. <a href="https://developer.android.com/studio/preview/install-preview">Learn more</a>.
</p>
<p>
To use the mentioned Android Emulator features make sure you are running at least Android Emulator v27.3+ downloaded via the Android Studio SDK Manager.
</p>
<p>
Please note, to ensure we maintain product quality, some of the features you saw in the canary channel like Navigation Editor are not enabled by default. To turn on canary release channel features go to <strong>File &#8594; Settings &#8594; Experimental &#8594; Editor &#8594; Enable Navigation Editor</strong>.
</p>
<p>
If you find a bug or issue, feel free to <a href="https://source.android.com/source/report-bugs#developer-tools">file an issue</a>. Connect with us -- the Android Studio development team &#8208; on our <a href="https://plus.google.com/103342515830390186255">Google+</a> page or on <a href="http://www.twitter.com/androidstudio">Twitter</a>.
</p>

          Android Things client library for Google Cloud IoT Core
         </a>
        </h2>
        <div class="post-header">
         <div class="published">

<img src="https://3.bp.blogspot.com/-_eD-YjwPRNU/Wyvr278WZ1I/AAAAAAAAFhg/c9cHpws01nggBl_PTgakQ8mkvVNHULR_QCLcBGAs/s1600/unnamed%2B%25288%2529image1.png" style="display:none">

<p>
<em>Posted by Wayne Piekarski, Developer Advocate for IoT <a href="https://google.com/+WaynePiekarski">+WaynePiekarski</a> <a href="https://twitter.com/waynepiekarski">@WaynePiekarski</a></em>
</p>
<p>
We're releasing a client library to make it easy to use Google Cloud IoT Core from Android Things devices. With just a few lines of code, you can easily connect to the IoT Core MQTT bridge, authenticate the device, publish device telemetry and state, subscribe to configuration changes, and handle errors and network outages.
</p>
<p>
<h2>What is Cloud IoT Core?</h2>
</p>
<p>
<a href="https://cloud.google.com/iot-core/">Cloud IoT Core</a> is a fully managed service on <a href="https://cloud.google.com">Google Cloud Platform</a> that allows you to easily and securely connect, manage, and ingest data from millions of globally dispersed devices. Cloud IoT Core, in combination with other services which make up Google's Cloud IoT platform, provides a complete solution for collecting, processing, analyzing, and visualizing IoT data in real time, to support improved operational efficiency, compliance, or revenue management. Android Things is designed to support everything from collecting telemetry data to powerful computer vision, audio processing, and machine learning applications, all on device, and using Cloud IoT Core, push your data into Google Cloud Platform for further analysis.
</p>
<p>
<a href="https://3.bp.blogspot.com/-_eD-YjwPRNU/Wyvr278WZ1I/AAAAAAAAFhg/c9cHpws01nggBl_PTgakQ8mkvVNHULR_QCLcBGAs/s1600/unnamed%2B%25288%2529image1.png" imageanchor="1" ><img border="0" data-original-height="722" data-original-width="1600" id="imgFull" src="https://3.bp.blogspot.com/-_eD-YjwPRNU/Wyvr278WZ1I/AAAAAAAAFhg/c9cHpws01nggBl_PTgakQ8mkvVNHULR_QCLcBGAs/s1600/unnamed%2B%25288%2529image1.png" /></a></p>
<p>
<h2>Cloud IoT Core client library</h2>
</p>
<p>
The Cloud IoT Core client library was designed to enable Android Things developers to get started with just a few lines of code. The client library handles the networking, threading, and message handling, implementing best practices for authentication, security, error handling, and offline operation.
</p>
<p>
Cloud IoT Core maintains a <a href="https://cloud.google.com/iot/docs/concepts/devices#device_registration">device registry</a> that keeps track of approved devices, and each device uses a public key to authenticate with the server. Android Things provides many features to support secure IoT applications, including a hardware-backed <a href="https://developer.android.com/training/articles/keystore">Android Keystore</a> that ensures cryptographic key material is protected. The client library supports both RSA and ECC keys, and implements the generation of JSON Web Tokens (JWTs) for authentication with Cloud IoT Core.
</p>
<p>
Once the connection is established, devices can publish their telemetry data to one or more buckets in the <a href="https://cloud.google.com/iot/docs/how-tos/mqtt-bridge#publishing_telemetry_events">telemetry topic</a>, as well as report their internal state to a separate <a href="https://cloud.google.com/iot/docs/how-tos/config/getting-state">device state topic</a>. The device state is intended to store information such as software versions or the number of working sensors. The telemetry messages are for all other data from the device, such as actual sensor measurements. Devices can also subscribe to <a href="https://cloud.google.com/iot/docs/concepts/devices#device_configuration">configuration changes</a> published from Cloud IoT Core.
</p>
<p>
Because IoT devices operate in the real world with poor wireless conditions, the client library provides extensive support for handling errors, and for caching and retransmitting events later. For developers requiring custom offline behavior, the library's queue is configurable and even replaceable. This provides detailed control over which events to save and the order in which they are sent when back online. 
</p>
<p>
<h2>Device provisioning and authentication with Android Things</h2>
</p>
<p>
The Cloud IoT Core client library is part of our overall vision for device provisioning and authentication with Android Things. To learn more about this, watch the video of our presentation from Google I/O 2018:
</p>
<p>
<iframe id="imgFull" width="560" height="315" src="https://www.youtube.com/embed/gkjV-TWLkIc" frameborder="0" allow="autoplay; encrypted-media" allowfullscreen></iframe></p>

<h2>Sample code</h2>
</p>
<p>
Getting started with the Cloud IoT Core client library is simple. You can simply add the following to the build.gradle file in your Android Things project:
</p>



<pre class="prettyprint">implementation 'com.google.android.things:cloud-iot-core:1.0.0'
</pre>


<p>
The library is also available as open source on <a href="https://github.com/androidthings/cloud-iot-core-androidthings">GitHub</a> if you prefer to build it yourself. We also have a <a href="https://github.com/androidthings/sensorhub-cloud-iot">sample</a> that shows how to implement a sensor hub on Android Things, collecting sensor data from connected sensors and publishing them to a Google Cloud IoT Pub/Sub topic.
</p>
<p>
It is easy to start using the client library in your own code. The following Kotlin example demonstrates how to create a new configuration and client based on your project.
</p>



<pre class="prettyprint">var configuration = IotCoreConfiguration.Builder().
                         .setProjectId("my-gcp-project")
                         .setRegistry("my-device-registry", "us-central1")
                         .setDeviceId("my-device-id")
                         .setKeyPair(keyPairObject)
                         .build()

var iotCoreClient = IotCoreClient.Builder()
              .setIotCoreConfiguration(configuration)
              .setOnConfigurationListener(onConfigurationListener)
              .setConnectionCallback(connectionCallback)
              .build()

iotCoreClient.connect()
</pre>


<p>
Next, you can publish telemetry information or device state, using the following Kotlin examples.
</p>



<pre class="prettyprint">private fun publishTelemetry(temperature: Float, humidity: Float) {
    // payload is an arbitrary, application-specific array of bytes
    val examplePayload = """{
        |"temperature" : $temperature,
        |"humidity": $humidity
        |}""".trimMargin().toByteArray()
    val event = TelemetryEvent(examplePayload, topicSubpath, TelemetryEvent.QOS_AT_LEAST_ONCE)
    iotCoreClient.publishTelemetry(event)
}

private fun publishDeviceState(telemetryFrequency: Int, enabledSensors: Array&lt;String>) {
    // payload is an arbitrary, application-specific array of bytes
    val examplePayload = """{
        |"telemetryFrequency": $telemetryFrequency,
        |"enabledSensors": ${enabledSensors.contentToString()}
        |}""".trimMargin().toByteArray()
    iotCoreClient.publishDeviceState(examplePayload)
}
</pre>

           <h2>
            What is Cloud IoT Core?
           </h2>
           <p>
            <a href="https://cloud.google.com/iot-core/">
             Cloud IoT Core
            </a>
            is a fully managed service on
            <a href="https://cloud.google.com">
             Google Cloud Platform
            </a>
            that allows you to easily and securely connect, manage, and ingest data from millions of globally dispersed devices. Cloud IoT Core, in combination with other services which make up Google's Cloud IoT platform, provides a complete solution for collecting, processing, analyzing, and visualizing IoT data in real time, to support improved operational efficiency, compliance, or revenue management. Android Things is designed to support everything from collecting telemetry data to powerful computer vision, audio processing, and machine learning applications, all on device, and using Cloud IoT Core, push your data into Google Cloud Platform for further analysis.
           </p>
           <p>
            <a href="https://3.bp.blogspot.com/-_eD-YjwPRNU/Wyvr278WZ1I/AAAAAAAAFhg/c9cHpws01nggBl_PTgakQ8mkvVNHULR_QCLcBGAs/s1600/unnamed%2B%25288%2529image1.png" imageanchor="1">
             <img border="0" data-original-height="722" data-original-width="1600" id="imgFull" src="https://3.bp.blogspot.com/-_eD-YjwPRNU/Wyvr278WZ1I/AAAAAAAAFhg/c9cHpws01nggBl_PTgakQ8mkvVNHULR_QCLcBGAs/s1600/unnamed%2B%25288%2529image1.png"/>
            </a>
           </p>
           <p>
           </p>
           <h2>
            Cloud IoT Core client library
           </h2>
           <p>
            The Cloud IoT Core client library was designed to enable Android Things developers to get started with just a few lines of code. The client library handles the networking, threading, and message handling, implementing best practices for authentication, security, error handling, and offline operation.
           </p>
           <p>
            Cloud IoT Core maintains a
            <a href="https://cloud.google.com/iot/docs/concepts/devices#device_registration">
             device registry
            </a>
            that keeps track of approved devices, and each device uses a public key to authenticate with the server. Android Things provides many features to support secure IoT applications, including a hardware-backed
            <a href="https://developer.android.com/training/articles/keystore">
             Android Keystore
            </a>
            that ensures cryptographic key material is protected. The client library supports both RSA and ECC keys, and implements the generation of JSON Web Tokens (JWTs) for authentication with Cloud IoT Core.
           </p>
           <p>
            Once the connection is established, devices can publish their telemetry data to one or more buckets in the
            <a href="https://cloud.google.com/iot/docs/how-tos/mqtt-bridge#publishing_telemetry_events">
             telemetry topic
            </a>
            , as well as report their internal state to a separate
            <a href="https://cloud.google.com/iot/docs/how-tos/config/getting-state">
             device state topic
            </a>
            . The device state is intended to store information such as software versions or the number of working sensors. The telemetry messages are for all other data from the device, such as actual sensor measurements. Devices can also subscribe to
            <a href="https://cloud.google.com/iot/docs/concepts/devices#device_configuration">
             configuration changes
            </a>
            published from Cloud IoT Core.
           </p>
           <p>
            Because IoT devices operate in the real world with poor wireless conditions, the client library provides extensive support for handling errors, and for caching and retransmitting events later. For developers requiring custom offline behavior, the library's queue is configurable and even replaceable. This provides detailed control over which events to save and the order in which they are sent when back online.
           </p>
           <p>
           </p>

            Getting started with the Cloud IoT Core client library is simple. You can simply add the following to the build.gradle file in your Android Things project:
           </p>

</pre>
           <p>
            The library is also available as open source on
            <a href="https://github.com/androidthings/cloud-iot-core-androidthings">
             GitHub
            </a>
            if you prefer to build it yourself. We also have a
            <a href="https://github.com/androidthings/sensorhub-cloud-iot">
             sample
            </a>
            that shows how to implement a sensor hub on Android Things, collecting sensor data from connected sensors and publishing them to a Google Cloud IoT Pub/Sub topic.
           </p>

            It is easy to start using the client library in your own code. The following Kotlin example demonstrates how to create a new configuration and client based on your project.
           </p>
           <pre class="prettyprint">var configuration = IotCoreConfiguration.Builder().
                         .setProjectId("my-gcp-project")

                         .build()

</pre>

          Better Biometrics in Android P
         </a>
        </h2>
        <div class="post-header">
         <div class="published">
          <span class="publishdate" itemprop="datePublished">
           21 June 2018
          </span>

<p>
<em>Posted by Vishwath Mohan, Security Engineer</em>
</p> 
<p>
To keep users safe, most apps and devices have an authentication mechanism, or a way to prove that you're you. These mechanisms fall into three categories: knowledge factors, possession factors, and biometric factors. <strong>Knowledge</strong> factors ask for something you know (like a PIN or a password), <strong>possession</strong> factors ask for something you have (like a token generator or security key), and <strong>biometric</strong> factors ask for something you are (like your fingerprint, iris, or face).
</p>
<p>
<a href="https://2.bp.blogspot.com/-PE6wsriQ8uA/WyrwwqW-FWI/AAAAAAAAFhE/o7MKqlPwqRQoimrSZb76bLlZfZgDpbLnACLcBGAs/s1600/%255BUNSET%255D%2B%252825%2529image1.png" imageanchor="1" ><img border="0" data-original-height="710" data-original-width="1600" id="imgFull" src="https://2.bp.blogspot.com/-PE6wsriQ8uA/WyrwwqW-FWI/AAAAAAAAFhE/o7MKqlPwqRQoimrSZb76bLlZfZgDpbLnACLcBGAs/s1600/%255BUNSET%255D%2B%252825%2529image1.png" /></a></p>
<p>
Biometric authentication mechanisms are becoming increasingly popular, and it's easy to see why. They're faster than typing a password, easier than carrying around a separate security key, and they prevent one of the most common pitfalls of knowledge-factor based authentication&#8212;the risk of <a href="https://en.wikipedia.org/wiki/Shoulder_surfing_(computer_security)">shoulder surfing</a>.
</p>
<p>
As more devices incorporate biometric authentication to safeguard people's private information, we're improving biometrics-based authentication in Android P by:
</p><ul>

<li>Defining a better model to measure biometric security, and using that to functionally constrain weaker authentication methods.
<li>Providing a common platform-provided entry point for developers to integrate biometric authentication into their apps.</li></ul>

<h2>A better security model for biometrics</h2>


<p>
Currently, biometric unlocks quantify their performance today with two metrics borrowed from  machine learning (ML): False Accept Rate (FAR), and False Reject Rate (FRR).
</p>
<p>
In the case of biometrics, FAR measures how often a biometric model accidentally classifies an incorrect input as belonging to the target user&#8212;that is, how often another user is falsely recognized as the legitimate device owner. Similarly, FRR measures how often a biometric model accidentally classifies the user's biometric as incorrect&#8212;that is, how often a legitimate device owner has to retry their authentication. The first is a security concern, while the second is problematic for usability.
</p>
<p>
Both metrics do a great job of measuring the accuracy and precision of a given ML (or biometric) model when applied to random input samples. However, because neither metric accounts for an active attacker as part of the threat model, they do not provide very useful information about its resilience against attacks.
</p>
<p>
In Android 8.1, we <a href="https://source.android.com/security/biometric/">introduced two new metrics</a> that more explicitly account for an attacker in the threat model: Spoof Accept Rate (SAR) and Imposter Accept Rate (IAR). As their names suggest, these metrics measure how easily an attacker can bypass a biometric authentication scheme. Spoofing refers to the use of a known-good recording (e.g. replaying a voice recording or using a face or fingerprint picture), while impostor acceptance means a successful mimicking of another user's biometric (e.g. trying to sound or look like a target user).
</p>
<h2>Strong vs. Weak Biometrics</h2>


<p>
We use the <a href="https://source.android.com/security/biometric/#metrics">SAR/IAR metrics</a> to categorize biometric authentication mechanisms as either strong or weak. Biometric authentication mechanisms with an SAR/IAR of 7% or lower are strong, and anything above 7% is weak. Why 7% specifically? Most fingerprint implementations have a SAR/IAR metric of about 7%, making this an appropriate standard to start with for other modalities as well. As biometric sensors and classification methods improve, this threshold can potentially be decreased in the future.
</p>
<p>
This binary classification is a slight oversimplification of the range of security that different implementations provide. However, it gives us a scalable mechanism (via the tiered authentication model) to appropriately scope the capabilities and the constraints of different biometric implementations across the ecosystem, based on the overall risk they pose.
</p>
<p>
While both strong and weak biometrics will be allowed to unlock a device, weak biometrics:
</p><ul>

<li>require the user to re-enter their primary PIN, pattern, password or a strong biometric to unlock a device after a 4-hour window of inactivity, such as when left at a desk or charger. This is in addition to the 72-hour timeout that is enforced for both strong and weak biometrics.
<li>are not supported by the forthcoming <a href="https://developer.android.com/reference/android/hardware/biometrics/BiometricPrompt">BiometricPrompt API</a>, a common API for app developers to securely authenticate users on a device in a modality-agnostic way.
<li>can't authenticate payments or participate in other transactions that involve a KeyStore auth-bound key.
<li>must show users a warning that articulates the risks of using the biometric before it can be enabled.</li></ul>

<p>
These measures are intended to allow weaker biometrics, while reducing the risk of unauthorized access.
</p>
<h2>BiometricPrompt API</h2>


<p>
Starting in Android P, developers can use the <a href="https://developer.android.com/reference/android/hardware/biometrics/BiometricPrompt">BiometricPrompt API</a> to integrate biometric authentication into their apps in a device and biometric agnostic way. BiometricPrompt only exposes strong modalities, so developers can be assured of a consistent level of security across all devices their application runs on. A support library is also provided for devices running Android O and earlier,  allowing applications to utilize the advantages of this API across more devices .
</p>
<p>
Here's a high-level architecture of BiometricPrompt. 
</p>
<p>
<a href="https://2.bp.blogspot.com/-gpo28lfiWEA/Wyrw4zvRbSI/AAAAAAAAFhI/p5LTm4vR64gpWMuZDqhewTlNd-U8tycRwCLcBGAs/s1600/sPsJBlMWpbA8FR6p3IwOZGwimage2.png" imageanchor="1" ><img border="0" data-original-height="364" data-original-width="624" id="imgFull" src="https://2.bp.blogspot.com/-gpo28lfiWEA/Wyrw4zvRbSI/AAAAAAAAFhI/p5LTm4vR64gpWMuZDqhewTlNd-U8tycRwCLcBGAs/s1600/sPsJBlMWpbA8FR6p3IwOZGwimage2.png" /></a></p>
<p>
The API is intended to be easy to use, allowing the platform to select an appropriate biometric to authenticate with instead of forcing app developers to implement this logic themselves. Here's an example of how a developer might use it in their app: 
</p>
<p>
<a href="https://1.bp.blogspot.com/-MYeVDoaqmu8/WyrxAiCwLQI/AAAAAAAAFhM/ScBGN8IGKWE9PLZDUEDrKvh0w1lx7uhzACLcBGAs/s1600/Screen%2BShot%2B2018-06-20%2Bat%2B5.23.50%2BPM.png" imageanchor="1" ><img border="0" data-original-height="352" data-original-width="1246" id="imgFull" src="https://1.bp.blogspot.com/-MYeVDoaqmu8/WyrxAiCwLQI/AAAAAAAAFhM/ScBGN8IGKWE9PLZDUEDrKvh0w1lx7uhzACLcBGAs/s1600/Screen%2BShot%2B2018-06-20%2Bat%2B5.23.50%2BPM.png" /></a></p>
<h2>Conclusion</h2>


<p>
Biometrics have the potential to both simplify and strengthen how we authenticate our digital identity, but only if they are designed securely, measured accurately, and implemented in a privacy-preserving manner.
</p>
<p>
We want Android to get it right across all three. So we're combining secure design principles, a more attacker-aware measurement methodology, and a common, easy to use biometrics API that allows developers to integrate authentication in a simple, consistent, and safe manner. 
</p>

             <img border="0" data-original-height="364" data-original-width="624" id="imgFull" src="https://2.bp.blogspot.com/-gpo28lfiWEA/Wyrw4zvRbSI/AAAAAAAAFhI/p5LTm4vR64gpWMuZDqhewTlNd-U8tycRwCLcBGAs/s1600/sPsJBlMWpbA8FR6p3IwOZGwimage2.png"/>
            </a>
           </p>
           <p>
            The API is intended to be easy to use, allowing the platform to select an appropriate biometric to authenticate with instead of forcing app developers to implement this logic themselves. Here's an example of how a developer might use it in their app:
           </p>
           <p>
            <a href="https://1.bp.blogspot.com/-MYeVDoaqmu8/WyrxAiCwLQI/AAAAAAAAFhM/ScBGN8IGKWE9PLZDUEDrKvh0w1lx7uhzACLcBGAs/s1600/Screen%2BShot%2B2018-06-20%2Bat%2B5.23.50%2BPM.png" imageanchor="1">
             <img border="0" data-original-height="352" data-original-width="1246" id="imgFull" src="https://1.bp.blogspot.com/-MYeVDoaqmu8/WyrxAiCwLQI/AAAAAAAAFhM/ScBGN8IGKWE9PLZDUEDrKvh0w1lx7uhzACLcBGAs/s1600/Screen%2BShot%2B2018-06-20%2Bat%2B5.23.50%2BPM.png"/>
            </a>
           </p>
           <h2>
            Conclusion
           </h2>
           <p>
            Biometrics have the potential to both simplify and strengthen how we authenticate our digital identity, but only if they are designed securely, measured accurately, and implemented in a privacy-preserving manner.
           </p>
           <p>
            We want Android to get it right across all three. So we're combining secure design principles, a more attacker-aware measurement methodology, and a common, easy to use biometrics API that allows developers to integrate authentication in a simple, consistent, and safe manner.
           </p>

         <a href="https://android-developers.googleblog.com/2018/06/grow-and-optimize-your-subscriptions.html" itemprop="url" title="Grow and optimize your subscriptions with new Google Play features">
          Grow and optimize your subscriptions with new Google Play features
         </a>
        </h2>
        <div class="post-header">
         <div class="published">
          <span class="publishdate" itemprop="datePublished">
           20 June 2018
          </span>
         </div>
        </div>
        <div class="post-body">
         <div class="post-content" itemprop="articleBody">
          <script type="text/template">
           <meta name="twitter:image" content="https://3.bp.blogspot.com/-dqp5KXwOe7E/WypeIxVSWgI/AAAAAAAAFgY/TqSejYz0564Jt1Y6dZVIsnMcvCucvAitQCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.36.50%2BPMimage1.png">
<img src="https://3.bp.blogspot.com/-dqp5KXwOe7E/WypeIxVSWgI/AAAAAAAAFgY/TqSejYz0564Jt1Y6dZVIsnMcvCucvAitQCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.36.50%2BPMimage1.png" style="display:none">

<p>
Posted by Larry Yang and Angela Ying, Product Managers, Google Play
</p>
<p>
Subscriptions on Google Play continue to see <a href="https://www.youtube.com/watch?v=RHwtJm3JPEo">huge growth</a>, with subscribers growing over 80% year over year. At <a href="https://www.youtube.com/watch?v=x1AYelepG6o">I/O 2018</a>, we announced several improvements we're making to the user experience to reduce barriers to subscription sign-up, and more tools to let you manage your business the way you want to.
</p>
<p>
<h2>More control for subscribers</h2>
</p>
<p>
While users derive a lot of value from their subscriptions, our research shows their fears of being "trapped" in a subscription without the ability to cancel or worry they'll lose track of how much they're spending create a hindrance to users signing up for your subscription apps. To address these fears, we recently launched a new <strong>subscriptions center</strong>, a one-stop shop for users to manage their subscriptions on Google Play. 
</p>
<p>
<a href="https://3.bp.blogspot.com/-dqp5KXwOe7E/WypeIxVSWgI/AAAAAAAAFgY/TqSejYz0564Jt1Y6dZVIsnMcvCucvAitQCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.36.50%2BPMimage1.png" imageanchor="1" ><img border="0" data-original-height="752" data-original-width="366" id="imgHalf" src="https://3.bp.blogspot.com/-dqp5KXwOe7E/WypeIxVSWgI/AAAAAAAAFgY/TqSejYz0564Jt1Y6dZVIsnMcvCucvAitQCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.36.50%2BPMimage1.png" style="width:30%" /></a>
</p>
<p>
Through the subscriptions center, users can:
</p><ul>

<li>View all of their subscriptions to see details and status
<li>Manage and update payment methods, including setting up a backup payment method
<li>Renew a subscription
<li>Restore a cancelled subscription
<li>Cancel a subscription</li></ul>

<p>
In addition, if a user cancels a subscription, we will now trigger a <strong>cancellation survey</strong> to give developers feedback as to why the user is cancelling. Currently you can see the data from the cancellation survey by <a href="https://developers.google.com/android-publisher/api-ref/purchases/subscriptions">querying our server side API</a>. 
</p>
<p>
<a href="https://4.bp.blogspot.com/-C0Qyh2NgSxU/WypeQAOaLII/AAAAAAAAFgg/NUxKo49VICYbGp4IriB_ee13bP5HD7f5gCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.38.51%2BPMimage2.png" imageanchor="1" ><img border="0" data-original-height="746" data-original-width="374" id="imgHalf" src="https://4.bp.blogspot.com/-C0Qyh2NgSxU/WypeQAOaLII/AAAAAAAAFgg/NUxKo49VICYbGp4IriB_ee13bP5HD7f5gCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.38.51%2BPMimage2.png" style="width:30%" /></a>
</p>
<p>
The new subscriptions center also has a "Get Started" link in the empty state that lets users discover subscription apps through curated and localized collections. 
</p>
<p>
<a href="https://1.bp.blogspot.com/-S4D6riHPaGY/WypeXoS3OrI/AAAAAAAAFgk/qWkrjsAc8F4daH1DYcOUW838LwPKhfGOgCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.50.19%2BPMimage3.png" imageanchor="1" ><img border="0" data-original-height="779" data-original-width="832" id="imgHalf" src="https://1.bp.blogspot.com/-S4D6riHPaGY/WypeXoS3OrI/AAAAAAAAFgk/qWkrjsAc8F4daH1DYcOUW838LwPKhfGOgCLcBGAs/s1600/Screen%2BShot%2B2018-06-19%2Bat%2B2.50.19%2BPMimage3.png" /></a>
</p>
<p>
With the launch of the subscriptions center, we're also launching new<strong> deep links</strong> you can use to direct your users to manage their subscriptions from your app, over email or via the web. To implement, use the package name and SKU to construct the deep link, and then add the deep link as a button or link from anywhere in your app. View the <a href="https://developer.android.com/google/play/billing/billing_subscriptions#deep-links-manage-subs">Android Developers website</a> for more information.
</p>
<p>
<h2>More control for you</h2>
</p>
<p>
In addition to creating a better experience for users, we're also rolling out new tools that give you <strong>more flexibility in managing your business</strong>. One of the features we've heard requested most is price changes. Coming soon, you can easily ask users to accept a price change via the Google Play Console without having to set up a completely new SKU. Google Play will notify users of the change via emails, push notifications and in-app messaging, and if by renewal date the user hasn't agreed, we'll cancel their subscription. <a href="https://docs.google.com/forms/d/1gKKtPHZsqzkUffbJGOM_DNMnr0a3N1_1dxXAbnUhGmM/edit">Sign up here</a> if you are interested in participating in the early access program. 
</p>
<p>
Other features we launched at I/O that help you better manage your subscription business include the ability to:
</p><ul>

<li><a href="https://developer.android.com/google/play/billing/billing_subscriptions#set-proratation-mode">Upgrade a subscription without changing the user's expiration date</a>
<li>Issue a partial refund from the Play Console 
<li>Refund specific subscription renewals, not just the latest one
<li>Use order IDs with the server-side API 
<li>Use the <a href="https://developers.google.com/android-publisher/api-ref/orders/refund">Refund API</a> with the Google Play Developer API </li></ul>

<p>
This is in addition to <a href="https://android-developers.googleblog.com/2018/01/faster-renewals-for-test-subscriptions.html">faster test renewals</a> and <a href="https://twitter.com/GooglePlayDev/status/964182774742769665">flexible intro pricing</a> we announced earlier this year.
</p>
<p>
To easily implement all of these, make sure you are using the <a href="https://developer.android.com/google/play/billing/billing_java_kotlin">Google Play Billing Library</a>, which launched <a href="https://developer.android.com/google/play/billing/billing_library_releases_notes">version 1.1</a> at I/O. The billing library is an abstraction layer on top of the AIDL file, and API updates are automatically picked up when you update your build dependency file the next time you compile your app. Price changes and upgrade/downgrade with the same expiration date are only available through the billing library. This will be the case for future launches as well.  
</p>
<p>
<h2>Better for everyone</h2>
</p>
<p>
We strongly believe that by building a great user experience, we build a high quality subscriber base. And by giving you tools and insights to better manage your business, you have the flexibility to do what is best for your business and your customers.  
</p>

            </a>
            we announced earlier this year.
           </p>
           <p>
            To easily implement all of these, make sure you are using the
            <a href="https://developer.android.com/google/play/billing/billing_java_kotlin">
             Google Play Billing Library
            </a>
            , which launched
            <a href="https://developer.android.com/google/play/billing/billing_library_releases_notes">
             version 1.1
            </a>
            at I/O. The billing library is an abstraction layer on top of the AIDL file, and API updates are automatically picked up when you update your build dependency file the next time you compile your app. Price changes and upgrade/downgrade with the same expiration date are only available through the billing library. This will be the case for future launches as well.
           </p>
           <p>
           </p>

            Better for everyone
           </h2>
           <p>
            We strongly believe that by building a great user experience, we build a high quality subscriber base. And by giving you tools and insights to better manage your business, you have the flexibility to do what is best for your business and your customers.

          Google Play security metadata and offline app distribution
         </a>
        </h2>
        <div class="post-header">

<em>Posted by James Bender, Product Manager, Google Play</em>
<p>
In December last year we <a href="https://android-developers.googleblog.com/2017/12/improving-app-security-and-performance.html">announced that we would be making updates to app security</a> to help verify product authenticity from Google Play. We are now adding a small amount of security metadata on top of APKs to verify that the APK was distributed by Google Play.
</p>
<p>
One of the reasons we're doing this is to help developers reach a wider audience, particularly in countries where peer-to-peer app sharing is common because of costly data plans and limited connectivity.
</p>
<p>
In the future, for apps obtained through Play-approved distribution channels, we'll be able to determine app authenticity while a device is offline, add those shared apps to a user's Play Library, and manage app updates when the device comes back online. This will give people more confidence when using Play-approved peer-to-peer sharing apps.
</p>
<p>
This also benefits you as a developer as it provides a Play-authorized offline distribution channel and, since the peer-to-peer shared app is added to your user's Play library, your app will now be eligible for app updates from Play.
</p>
<p>
No action is needed by developers or by those who use your app or game. We're adjusting Google Play's maximum APK size to take into account the small metadata addition, which is inserted into the <a href="https://source.android.com/security/apksigning/v2">APK Signing Block</a>. In addition to improving the integrity of Google Play's mobile app ecosystem, this metadata will also present new distribution opportunities for developers and help more people keep their apps up to date.

         <a href="https://android-developers.googleblog.com/2018/06/an-update-on-non-sdk-restrictions-in.html" itemprop="url" title="An Update on non-SDK restrictions in Android P">
          An Update on non-SDK restrictions in Android P
         </a>
        </h2>
        <div class="post-header">
         <div class="published">
          <span class="publishdate" itemprop="datePublished">
           12 June 2018
          </span>
         </div>
        </div>
        <div class="post-body">

           <meta name="twitter:image" content="https://1.bp.blogspot.com/-4SsYBsjXSI8/Wx_hi64XjzI/AAAAAAAAFe0/_KyjAeQW7FoPWczm248vN9FyHPC2BGwRQCLcBGAs/s1600/blog_image.png">
<img src="https://1.bp.blogspot.com/-4SsYBsjXSI8/Wx_hi64XjzI/AAAAAAAAFe0/_KyjAeQW7FoPWczm248vN9FyHPC2BGwRQCLcBGAs/s1600/blog_image.png" style="display:none">

<p>
<em>Posted by David Brazdil and Nicolas Geoffray, Software Engineers</em>
</p>
<p>
In Android, we care immensely about providing the best experience to our users and our developers. With each OS release, new features enable you to provide amazing experiences for users; however, we noticed that some app developers have been using non-SDK interfaces, which leads to increased crashes for users and emergency rollouts for developers. We want to do better and need your help to ensure that Android is stable with each new OS. 
</p>
<p>
Three months ago, we announced our plans to <a href="https://android-developers.googleblog.com/2018/02/improving-stability-by-reducing-usage.html">start restricting the usage of non-SDK interfaces in Android P</a>. We know these restrictions can impact your release workflow, and we want to make sure you have the tools to detect usage of non-SDK interfaces, as well as time in your planning for adjusting to the new policies and for giving us feedback.
</p>
<p>
In the Developer Preview and Beta 1, we have provided ways for you to see the impact of these restrictions on your app. In the <a href="https://developer.android.com/preview/">Developer Preview</a>, use of restricted APIs show up in logs and toast messages, and in Beta 1, we provided a <a href="https://developer.android.com/reference/android/os/StrictMode">StrictMode</a> policy that allows you to programmatically find these restrictions and do your own logging. For example:
</p>
<p>
<a href="https://1.bp.blogspot.com/-4SsYBsjXSI8/Wx_hi64XjzI/AAAAAAAAFe0/_KyjAeQW7FoPWczm248vN9FyHPC2BGwRQCLcBGAs/s1600/blog_image.png" imageanchor="1" ><img border="0" data-original-height="425" data-original-width="1334" id="imgFull" src="https://1.bp.blogspot.com/-4SsYBsjXSI8/Wx_hi64XjzI/AAAAAAAAFe0/_KyjAeQW7FoPWczm248vN9FyHPC2BGwRQCLcBGAs/s1600/blog_image.png" style="width: 100%" /></a></p>
<p>
We understand there can be multiple reasons apps want to use non-SDK interfaces, and ensuring your app will continue to work in Android P is important to us. Many of you have explained your use cases through our <a href="https://issuetracker.google.com/issues/new?component=328403&template=1027267">issue tracker</a> (thank you!), and for most of these requests, we have lifted the restrictions on specific non-SDK interfaces for Android P by adding them to the <a href="https://android.googlesource.com/platform/frameworks/base/+/85c81cb24c5d71fc79dea3ba498d1a509895bb01/config/hiddenapi-p-light-greylist.txt">greylist</a>. In addition, our team has conducted static analysis on millions of apps and processed thousands of automated reports from internal and external beta testers. Through this analysis, we have identified additional non-SDK interfaces that apps rely on and added them to the greylist. For everything on the greylist, we will be investigating public SDK alternatives for future releases. However, it's possible we may have missed some non-SDK interface uses, so we have made the majority of them available for apps whose target SDK is Android Oreo or below. 
</p>
<p>
In summary, apps running on Android P will be subject to restricted usage of non-SDK interfaces. If you are targeting Android P, the <a href="https://android.googlesource.com/platform/frameworks/base/+/85c81cb24c5d71fc79dea3ba498d1a509895bb01/config/hiddenapi-p-light-greylist.txt">greylist</a> shows the non-SDK interfaces that will still be available, while all other non-SDK interfaces will not be accessible. If you are targeting Android Oreo or below, most restrictions will not apply, but you will get logcat warnings if you are accessing non-SDK interfaces that are not in the greylist (note that users don't see such warnings).
</p>
<p>
Try out our new <a href="https://android-developers.googleblog.com/2018/06/android-p-beta-2-and-final-apis.html">Beta 2 release</a> and use <a href="https://developer.android.com/reference/android/os/StrictMode">StrictMode</a> to detect your non-SDK interfaces usage. You should expect Beta 2 to closely match what the final release will implement for restricting non-SDK interface usages. Also, please take a look at our new <a href="https://developer.android.com/preview/restrictions-non-sdk-interfaces">FAQ</a>, which we hope will answer any questions you have around the feature. If not, <a href="https://issuetracker.google.com/issues/new?component=328403&template=1027267">let us know</a>!
</p>

            shows the non-SDK interfaces that will still be available, while all other non-SDK interfaces will not be accessible. If you are targeting Android Oreo or below, most restrictions will not apply, but you will get logcat warnings if you are accessing non-SDK interfaces that are not in the greylist (note that users don't see such warnings).
           </p>
           <p>
            Try out our new
            <a href="https://android-developers.googleblog.com/2018/06/android-p-beta-2-and-final-apis.html">
             Beta 2 release
            </a>
            and use
            <a href="https://developer.android.com/reference/android/os/StrictMode">
             StrictMode
            </a>
            to detect your non-SDK interfaces usage. You should expect Beta 2 to closely match what the final release will implement for restricting non-SDK interface usages. Also, please take a look at our new
            <a href="https://developer.android.com/preview/restrictions-non-sdk-interfaces">
             FAQ
            </a>
            , which we hope will answer any questions you have around the feature. If not,
            <a href="https://issuetracker.google.com/issues/new?component=328403&amp;template=1027267">
             let us know
            </a>
            !
           </p>
          </noscript>
