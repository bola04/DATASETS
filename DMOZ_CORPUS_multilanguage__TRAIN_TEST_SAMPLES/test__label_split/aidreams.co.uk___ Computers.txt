           <h2>
            <a href="http://aidreams.co.uk/forum/index.php?topic=12949">
             S-Expression Query Language
            </a>
            in General Project Discussion
           </h2>
           <p class="block-desc">
            <span class="bbc_size" style="font-size: 1.45em;">
             Binary Structure Calculus
            </span>
            <br/>
            <br/>
            Recently, I was exploring some
            <a class="bbc_link" href="https://en.wikipedia.org/wiki/S-expression" target="_blank">
             s-expression
            </a>
            notation known from LISP breed of languages. LISP languages are usually used for AI (a part of OpenCog is based on it). What I was trying to achieve is having an algebra of duality. For example, we define summation function and automatically derive difference function. Or we define multiply function and automatically derive division function. Or we define emergence, and conclude consequence from specifying cause, or conclude cause from specifying consequence. In general, we define a function, and we could query it in both directions, from parameters to result, or from result to parameters. I believe it has something to do with a
            <a class="bbc_link" href="https://simple.wikipedia.org/wiki/P_versus_NP" target="_blank">
             million dollar prized P = NP? question
            </a>
            .
            <br/>
            <br/>
            All of this should nicely fit with math, logic, knowledge base querying, theorem proving, and it could find a use in any science field, I believe.
            <br/>
           </p>
           <hr/>
           This is syntax in BNF notation - this is the whole language, there is nothing else:
           <br/>
           <div class="codeheader">
            Code:
            <a class="codeoperation" href="javascript:void(0);" onclick="return smfSelectText(this);">
             [Select]
            </a>
           </div>
           <code class="bbc_code prettyprint lang-aidreams">
            start ::= e    //s-expression
            <br/>
            <br/>
            e ::= a        // atom
            <br/>
            | a e      // sequencing
            <br/>
            | (e)      // grouping
            <br/>
            <br/>
            a ::= c        // constant
            <br/>
            | {q}      // query
            <br/>
            <br/>
            q ::= a q      // querying to the right of a
            <br/>
            | q a      // querying to the left of a
            <br/>
            | ?        // return query result
            <br/>
            | !        // return query complement
            <br/>
            | (q)      // grouping
            <br/>
           </code>
           <hr/>
           These are seven semantic rules expressed in a kind of
           <a class="bbc_link" href="https://en.wikipedia.org/wiki/Sequent_calculus" target="_blank">
            Sequent Calculus
           </a>
           :
           <br/>
           <span style="text-shadow: gray 1px 1px 1px">
            <span class="bbc_size" style="font-size: 8pt;">
             <tt class="bbc_tt">
              <br/>
              <br/>
              constant c ϵ Γ
              <br/>
              ────────────────────── (1)
              <br/>
              Γ ⊢ c
              <br/>
              <br/>
              <br/>
              Γ ⊢ c1     Γ ⊢ c2
              <br/>
              ───────────────────────── (2)
              <br/>
              Γ ⊢ c1 c2
              <br/>
              <br/>
              <br/>
              <br/>
              ───────────────────────── (3)
              <br/>
              Γ, {σ} ⊢ Σ(σ) → Γ
              <br/>
              <br/>
              <br/>
              Σ(c σ) → c c1     Σ(c σ) → c c2     ...
              <br/>
              ─────────────────────────────────────────────── (4)
              <br/>
              (Σ(σ) → c1) ∧ (Σ(σ) → c2) ∧ ...
              <br/>
              <br/>
              <br/>
              Σ(σ c) → c1 c     Σ(σ c) → c2 c     ...
              <br/>
              ─────────────────────────────────────────────── (5)
              <br/>
              (Σ(σ) → c1) ∨ (Σ(σ) → c2) ∨ ...
              <br/>
              <br/>
              <br/>
              Σ(?) → σ
              <br/>
              ──────────────── (6)
              <br/>
              σ
              <br/>
              <br/>
              <br/>
              Σ(!) → σ
              <br/>
              ─────────────────────── (7)
              <br/>
              complement of σ
              <br/>
              <br/>
             </tt>
            </span>
           </span>
           <br/>
           <br/>
           Rules 1-2 are sequence introducing rules. Rules 3-7 are sequence query mechanism. (3) is converting a query to intermediate sigma form. (4) and (5) are sigma tree traversal inference. (6) and (7) are converting terminal sigma form to usable end result.
           <br/>
           <hr/>
           Basically, Binary Structure Calculus is a syntax of s-expressions plus syntax of querying over s-expressions. Query can simply be
           <span class="bbc_font" style="font-family: courier;">
            {x ?}
           </span>
           for reaching element right to matched x, or
           <span class="bbc_font" style="font-family: courier;">
            {? x}
           </span>
           for reaching element left to matched x. Of course, queries can be nested to reach complex structure elements. And that's all there is, and it seems enough.
           <br/>
           <br/>
           You'll be surprised how expressive this combination might be. More about it soon, I'm preparing a whitepaper.

             UIZero, the simple full-width cards-stack
            </a>
            in Human Computer Interaction
           </h2>
           <p class="block-desc">
            So I'm working for an interface for my next project (the thinkbot thing).
            <br/>
            <br/>
            I thought I would use Goldenlayout, but I decided to roll my own, very simple design.
            <br/>
            <br/>
            <a class="bbc_link" href="http://t3js.org/#core-philosophy" target="_blank">
             T3js
            </a>
            allows scalability through modularity. An application is made of isolated modules. Modules are associated with a DOM node they're in charge of. My understanding of this is translated in UIZero: simple DIVs, that's it.
            <br/>
            <br/>
            They say the
            <a class="bbc_link" href="https://www.qwant.com/?q=ui%20cards%20paradigm&amp;t=web" target="_blank">
             cards paradigm
            </a>
            is future of UI. Well, if we're gonna do
            <em>
             technical
            </em>
            UI, cards don't have to be all shiny and new (like a virgin). Instead, they need to be full-width, and we should be able to show/hide them easily. That's what UIZero does, and it does only that.
            <br/>
            <br/>
            <a class="bbc_link" href="http://thinkbots.are.free.fr/ProjectUIZero/" target="_blank">

            <a class="bbc_link" href="http://news.mit.edu/2018/mit-csail-launches-new-initiative-financial-technology-0726" target="_blank">
             CSAIL launches new initiative for financial technology
            </a>
            <br/>
            <strong>
             26 July 2018, 3:40 pm
            </strong>
            <br/>
            <br/>
            Recent advances in automating and digitizing financial services has largely changed how we use technology to make fiscal decisions. As infrastructure and operations are changing rapidly — recent reports show
            <a class="bbc_link" href="http://home.kpmg.com/xx/en/home/media/press-releases/2018/02/global-fintech-funding-tops-us-31b-for-2017-fueled-by-us-in-q4-kpmg-pulse-of-fintech-report.html" target="_blank">
             $31 billion was invested
            </a>
            in the sector in 2017 — reinventing global financial technology poses many challenges and potential rewards.
            <br/>
            <br/>
            To address these issues, MIT’s Computer Science and Artificial Intelligence Laboratory (CSAIL) recently launched a research-industry collaboration focused on creating financial technologies that will be able to open up new business models, gain new data insights, and improve security.
            <br/>
            <br/>
            The initiative will span topics that include artificial intelligence, cryptocurrencies, blockchain foundation and applications, machine learning, multi-party computation for superior security and privacy, data management and analytics, natural language processing, and cyber-risk management, among others.
            <br/>
            <br/>
            MIT professors Andrew Lo, Silvio Micali, and Shafi Goldwasser are leading the new initiative, which is called FinTech@CSAIL. A select group of companies, including ANT Financial, Citi, London Stock Exchange Group, Nasdaq, Ripple, Ryan LLC, and State Street, will work with CSAIL researchers to inform new lines of impactful research and develop innovative real-world applications.
            <br/>
            <br/>
            “A financial company’s success in this sector is built on trust, security, value, and efficiency,” says FinTech Executive Director Lori Glover. “FinTech holds promise not only for verified transaction systems such as blockchain, but also for technologies involving AI, security, data analytics, trust verification, risk management and privacy advances as well.”
            <br/>
            <br/>
            CSAIL researchers will focus on several key areas across the lab, including cybersecurity and secure computation, natural language processing, robotics, and replacement technologies and legacy systems.
            <br/>
            <br/>
            Lo, Micali, and Goldwasser are guiding the effort with a unique combined expertise focused on economics, computer science and cybersecurity.
            <br/>
            <br/>
            Lo’s research centers on using computational tools to make better financial decisions, including financial engineering and risk management, trading technology and market microstructure, as well as creating computational models for individual risk preferences, financial markets, and intelligence.
            <br/>
            <br/>
            During his remarks at the Wednesday launch event, Lo talked about the interplay of finance and technology and how they have impacted the way that AI systems are used.
            <br/>
            <br/>
            Specifically, he discussed early efforts in AI that focused on very specific problems and tasks, like playing chess, and how they aimed to covered as many different scenarios for each problem as possible. He said that this approach was a timely and expensive endeavour that was ineffective for financial models.
            <br/>
            <br/>
            “We’ve largely moved away from AI as expert systems to simple algorithms and more complex data, which is much closer to real human intelligence,” Lo said. “We need theories and algorithms on how people actually behave, and larger datasets where we can find meaningful trends and patterns. Unless we truly understand human behavior, we won’t we be able to make financial technologies that work.”
            <br/>
            <br/>
            Micali and Goldwasser, who are both focused on security, have pioneered research at the foundation of current and emerging fintech innovations. Both are winners of the A.C.M. Turing Award, often described as “the Nobel of computing.”
            <br/>
            <br/>
            Micali and Goldwasser’s research efforts focus on enhancing information security and the privacy and correctness of computation and data. They have been instrumental in developing systems like zero-knowledge proofs (methods for authentication that don’t require passwords), and secure multi-party computation, which are both used in many cybersecurity systems and financial applications.
            <br/>
            <br/>
            “We believe in transferring technology to society, which is a value of not only fintech but CSAIL as well,” said Micali. “As this sector rapidly matures, we need to make technologies that can keep up with that pace.”
            <br/>
            <br/>
            FinTech@CSAIL builds on the lab's other initiatives, which include
            <a class="bbc_link" href="http://cybersecurity.csail.mit.edu" target="_blank">
             Cybersecurity@CSAIL
            </a>
            and
            <a class="bbc_link" href="http://stl.csail.mit.edu/" target="_blank">
             SystemsThatLearn@CSAIL
            </a>
            .
            <br/>
            <br/>
            “With a legacy that dates back to the originators of artificial intelligence and many computational techniques in use today, CSAIL is at the leading edge of research and new technologies that can benefit the entire financial ecosystem,” said Lo.

           The obviousness of the information represented here is striking! This is an absolutely wild piece of data, and yet it makes perfect sense to a human developer. For a long time, as soon as I was planning to manipulate knowledge, I used to think "cyc-style ontology". Today, I'm wondering after all, what's wrong with messy data? Messy data's ok!!!
           <br/>
           <br/>
           I'm working on a framework that would allow a hobbyist developer to shape a mind as easily as a chatbot. Well, I'm not saying that building up a good chatbot is easy (Mitsuku is probably a very complex project for example). Still, it's easier to make a chatbot than to make a Schwarzenegger-style metal bones AGI. I'd like a thinkbot developer to feel like working on a chatbot: simple tech, no pressure, hobby, relax, enjoy.
           <br/>
           <br/>
           The only real problem is name collision if you're dealing with huge knowledge base. The answer to this is simply not to use named entities. Knowledge is just a list of objects. Objects have keys which lead to values. But they have no IDs, hence no name collision.
