      <h4>
       <span class="mw-headline" id="Information_for_volunteers">
        Information for volunteers
       </span>
      </h4>
      <p>
       If you are a developer and have some code-project for the Droid family of smartphones(e.g. Milestone) - enjoy us on
       <a class="external text" href="http://gitorious.org/+droid-developers" rel="nofollow">
        Gitorious
       </a>
      </p>
      <p>
       Even if you're not the technical type, you too can help us mod the Milestone by participating in the
       <a href="/wiki/Custom_recovery:pr_attack" title="Custom recovery:pr attack">
        PR campaign to force Motorola to unlock it
       </a>
       .
      </p>
      <p>
       If you're technical type - see our
       <a href="/wiki/Roadmap" title="Roadmap">
        Roadmap
       </a>
       and progress in our
       <a href="/wiki/Projects" title="Projects">
        Projects
       </a>
       .
      </p>
      <p>
       See the
       <a href="/wiki/Content" title="Content">
        content index here
       </a>
       .

           <td style="vertical-align:top">
            <p>
             The
             <a href="/wiki/Modes" title="Modes">
              recovery image
             </a>
             hasn't yet been
             <a class="external text" href="http://android-dls.com/wiki/index.php?title=HOWTO:_Unpack,_Edit,_and_Re-Pack_Boot_Images" rel="nofollow">
              modified
             </a>
             due to our current impossibility of controlling
             <a href="/wiki/Booting_chain" title="Booting chain">
              the boot process
             </a>
             . We cannot alter the boot process so far because there seems to be a digital signature on each of its components. It seems the bootloader (mbm in particular (neither lbl nor mbmloader access the CDT). See
             <a href="/wiki/Booting_chain" title="Booting chain">
              here
             </a>
             and
             <a class="external text" href="https://opensource.motorola.com/sf/discussion/do/listPosts/projects.milestone/discussion.general_comments.topc2012?_pagenum=2" rel="nofollow">
              here
             </a>
             .) uses the
             <a href="/wiki/CDT" title="CDT">
              cdt partition table
             </a>
             to check if the recovery has been signed correctly. If not, the recovery won't start at all and the
             <a href="/wiki/Modes" title="Modes">
              bootloader mode
             </a>
             shows instead of it.
            </p>

            <p>
             We have running RTXC OS on
             <a href="/wiki/Wrigley_3G" title="Wrigley 3G">
              Wrigley 3G
             </a>
             modem, which consist from ARM core and
             <a href="/wiki/TMS320C55x%2B" title="TMS320C55x+">
              TMS320C55x+
             </a>
             DSP core
 Our main problem, that Motorola using non-standard RIL, which partially implemented on AP side.
 Also, our second problem, that
             <a href="/wiki/TMS320C55x%2B" title="TMS320C55x+">
              TMS320C55x+
             </a>
             is closed platform, and no datasheets for it available.
 It very different from original
             <a class="new" href="/index.php?title=TMS320C55x&amp;action=edit&amp;redlink=1" title="TMS320C55x (page does not exist)">

             </a>
             architecture and have other opcodes.
 But
             <a class="external text" href="http://rada.re/" rel="nofollow">
              radare2
             </a>
             utility have support for this platform and can do disassembly and simple analysis (you must use version from git).
 We have also
             <b>
              asm55p
             </b>
             utility from TI, which can produce binary from TMS320 assembler.
            </p>

      <h2>
       <span class="mw-headline" id="2ndboot">

          2ndboot
         </a>
        </b>
       </span>
      </h2>
      <p>
       A minature bootloader that is called from the original kernel and boots custom one. As of 11/10/2012,
       <b>
        czechop
       </b>
       created a patch to keep Wrigley 3G modem working under the child kernel (when called at “sh hijack” time). No issues on Motorola Milestone with the child kernel.
      </p>
      <h2>
       <span class="mw-headline" id="Vulnerability_hunting">
        <b>
         <a href="/wiki/Vulnerability_hunting" title="Vulnerability hunting">
          Vulnerability hunting
         </a>
        </b>
       </span>
      </h2>
      <p>
       As far as we know now this attack is, if not a waste of time, at least a very long shot. The idea is that reverse engineering the boot ROM, mbmloader and/or mbm might allow us to find some exploitable vulnerability in order to gain control of the boot process. Since we already have the source code for lbl, it might be useful too. Mike Baker([mbm]) has written a
       <a class="external text" href="http://pastebin.ca/raw/1833228%7Cuser" rel="nofollow">
        mode memory dumper
       </a>
       and dumped Droid public ROM. As we found - all roms for omap3430 are identical. Same situation for the omap3630. See here:
       <a href="/wiki/Booting_chain" title="Booting chain">
        Boot chain
       </a>
      </p>
      <h2>
       <span class="mw-headline" id="Open_Recovery">
        <b>
         <a href="/wiki/Open_recovery" title="Open recovery">
          Open Recovery
         </a>
        </b>
       </span>
      </h2>
      <p>
       Uses the payload exploit to start the custom recovery application. Supports rooting the phone from menu, as well as taking backups and flashing unsigned update *.zip files. Also runs ADB.
      </p>
      <h2>
       <span class="mw-headline" id="2ndinit">
        <b>
         <a href="/wiki/2ndinit" title="2ndinit">
          2ndinit
         </a>
        </b>
       </span>
      </h2>
      <p>
       This thing basically inject code to /init to "restart itself" allowing you to use custom init binary and init.rc scripts without side effects.
      </p>
