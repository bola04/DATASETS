        <h3>
         <a href="https://doughughes.net/2018/07/06/ruby-void-value-expression-errors-explained/" rel="bookmark">
          Ruby Void Value Expression Errors Explained
         </a>
        </h3>
        <div class="post-date">
         <span>
          July 6, 2018
         </span>
        </div>
       </div>
       <div class="entry">
        <p>
         Today I was doing a quick code review on a pull request and noticed that
         <em>
          all
         </em>
         of our test suite had failed CI. This was surprising since the change made was so small and looked so reasonable.
        </p>
        <p>
         This is (more or less) what the method had been:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def query
   @query ||= begin
     query = SomeModel.where(...)
     query = query.addressed if addressed
     query
   end
 end
 </pre>
        <p>
         The details of
         <code>
          addressed
         </code>
         are irrelevant to this article, so don’t worry about them. I’m leaving it here to illustrate that the
         <code>
          query
         </code>
         variable may have been updated under some circumstances.
        </p>
        <p>
         The method had been updated like this in the PR:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def query
   @query ||= begin
     query = SomeModel.where(...)
     query = query.addressed if addressed
     return query.to_a
   end
 end
 </pre>
        <p>
         So, all that happened was that the last line of the
         <code>
          begin
         </code>
         block had been changed from
         <code>
          query
         </code>
         to
         <code>
          return query.to_a
         </code>
         . It’s not important to get into the reasons why the query was being turned into an array, it just was.
        </p>
        <p>
         However, this caused all of our tests to fail with a “void value expression (SyntaxError)”.
        </p>
        <p>
         At first I didn’t see what was going on, but, after some research, I figured it out. Long story short, this would work just fine without the
         <code>
          return
         </code>
         keyword.
        </p>
        <p>
         It seems natural to think of a
         <code>
          begin
         </code>
         block as an inline method that has its own return value, but, in reality, it’s an expression that evaluates to a value. There’s a subtle difference. Consider this expression:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 5 + 5
 </pre>
        <p>
         This expression
         <em>
          evaluates
         </em>
         to 10. It’s not a method that returns a value. If you run that in a Ruby console you’ll see as much.
        </p>
        <p>
         This is not a valid expression in Ruby, outside the context of a method:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 return 5 + 5
 </pre>
        <p>
         Running this in a console gives you an “unexpected return” error.
        </p>
        <p>
         Next, let’s consider this method:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo
 	5 + 5
 end
 </pre>
        <p>
         This method, when evaluated, returns the result of evaluating the expression
         <code>
          5 + 5
         </code>
         . The reason it returns this is because Ruby assumes that the last expression evaluated in any method is its return value, unless something else is returned first.
        </p>
        <p>
         The above is exactly the same as this next example, we’re just being explicit now:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo
 	return 5 + 5
 end
 </pre>
        <p>
         It’s important to know that the
         <code>
          return
         </code>
         keyword immediately terminates execution of the method and returns the specified value (or
         <code>
          nil
         </code>
         ). Here’s an example with an early return:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo
 	return 75
 	5 + 5
 end
 </pre>
        <p>
         As you might expect, this returns
         <code>
          75
         </code>
         due to the early return terminating execution and returning the specified value.
        </p>
        <p>
         Now let’s consider variable assignments:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo
 	a = 5 + 5
 	return a
 end
 </pre>
        <p>
         Running this method returns
         <code>
          10
         </code>
         . Keep in mind that the
         <code>
          return
         </code>
         statement is immediately terminating the method and returning the value of
         <code>
          a
         </code>
         .
        </p>
        <p>
         Here’s another variable assignment example:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo(bool)
 	if bool
 		a = 5 + 5
 	else
 		a = 75
 	end
 	return a
 end
 </pre>
        <p>
         In this example we’re setting
         <code>
          a
         </code>
         to 10 or 75 depending on a conditional statement and then returning
         <code>
          a
         </code>
         . This is one way to write this logic. We could also have written it this way:
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo(bool)
 	a = if bool
 		5 + 5
 	else
 		75
 	end
 	return a
 end
 </pre>
        <p>
         This works because
         <code>
          if
         </code>
         statements in Ruby (like
         <code>
          case
         </code>
         statements, blocks, etc)
         <em>
          evaluate
         </em>
         to a value. It’s tempting to think of them as returning a value, but that’s not what’s going on. If they
         <em>
          returned
         </em>
         , the
         <code>
          foo
         </code>
         method would immediately terminate and return the value of the statement. To put it another way,
         <code>
          a
         </code>
         is set to the result of evaluating the entire conditional statement.
        </p>
        <p>
         But what about this example?
        </p>
        <pre class="brush: ruby; title: ; notranslate" title="">
 def foo(bool)
 	a = if bool
 		5 + 5
 	else
 		return 75
 	end
 	return a
 end
 </pre>
        <p>
         If you try to paste this into a Ruby console you’ll get a “void value expression” error. But why? Isn’t it effectively the same as the example above? Nope! Ruby sees the
         <code>
          a =
         </code>
         and knows it’s expected to assign something to
         <code>
          a
         </code>
         . The “something” being assigned is the result of evaluating the expression to the right of the
         <code>
          =
         </code>
         operator.
        </p>
        <p>
         It’s pretty clear what this should be if
         <code>
          bool
         </code>
         is
         <code>
          true
         </code>
         , but what if it’s
         <code>
          false
         </code>
         ? Our instinct might be to assume that the
         <code>
          if
         </code>
         statement returns a value, but, as we determined above, that’s not how
         <code>
          return
         </code>
         works.
        </p>
        <p>
         Effectively, Ruby sees that it could be in the middle of setting
         <code>
          a
         </code>
         and be interrupted to return
         <code>
          75
         </code>
         from
         <code>
          foo
         </code>
         , effectively leaving
         <code>
          a
         </code>
         abandoned. That is, the expression to the right of the
         <code>
          =
         </code>
         operator either evaluates to a value or not at all (IE: a
         <em>
          void value
         </em>
         ). Ruby doesn’t like this and raises a “void value expression” error.
        </p>
        <p>
         So, looking back at the
         <code>
          query
         </code>
         method above, it’s trying to set
         <code>
          @query
         </code>
         to the result of evaluating the
         <code>
          begin ... end
         </code>
         <em>
          expression
         </em>
         . But, before it can do so, it encounters a
         <code>
          return
         </code>
         statement that wants to immediately exit the
         <code>
          query
         </code>
         method, which Ruby dislikes.
        </p>
        <p>
         The solution to this problem is simple: Just remove the
         <code>
          return
         </code>
         keyword. Since Ruby assumes that a statement’s value is the last line of code evaluated, Ruby will set
         <code>
          @query
         </code>
         to the result of evaluating
         <code>
          query.to_a
         </code>
         .
        </p>
       </div>

        <h3>
         <a href="https://doughughes.net/2017/03/25/testing-for-perfect-numbers-in-java-with-streams/" rel="bookmark">
          Testing for Perfect Numbers in Java with Streams
         </a>
        </h3>
        <div class="post-date">
         <span>
          March 25, 2017
         </span>
        </div>
       </div>
       <div class="entry">
        <p>
         I was challenged recently to write a Java method that could validate whether a provided number was “perfect” or not. A
         <a href="https://en.wikipedia.org/wiki/Perfect_number">
          perfect number
         </a>
         is one where the sum of its divisors (excluding itself) add up to the number. For example, 28 is a perfect number because it’s divisors are: 1, 2, 4, 7, and 14. If you add these up you get 28, the original number.
        </p>
        <p>
         However, consider 16. It’s divisors are: 1, 2, 4, 4, and 8. If you add these up you get 19. Therefore, 16 is not perfect.
        </p>
        <p>
         This calculation was to be done via a static method that accepted an argument
         <code>
          num
         </code>
         . The method was to return a boolean indicating whether the number provided was perfect or not.
        </p>
        <p>
         My first attempt was naive. I stared by using a for loop to loop backwards from one less than the number being tested to 1. I tested each step to determine if the number was divisible by the step. If so, I collected both numbers into a tuple which I collected into a HashSet to remove duplicates. I then iterated over the set and summed the values in the tuple and added 1. It worked, but was considerably inefficient.
        </p>
        <p>
         It was pointed out to me that, because I was looping backwards, I was doing two times the amount of work needed. If I looped forward to
         <em>
          half
         </em>
         the number being tested I could simply create a sum of the valid divisors. Understanding this, I decided to try a few other approaches…
        </p>
        <p>
         First, I wanted to see if I could make a one-line solution to this puzzle. In particular, I wanted to experiment with streams of numbers in Java. My first attempt used a stream of integers. I reduced this to those which were valid divisors of the number being tested and summed the resulting set. If this was equal to the number, the validation passed. If not, it didn’t. This is the code:
        </p>
        <pre class="brush: java; title: ; notranslate" title="">
 public static boolean reduceIsPerfect(long num) {
   return LongStream.rangeClosed(2, num / 2)
     .reduce(1, (sum, test) -&gt; num % test == 0 ? sum + test : sum) == num;
 }
 </pre>
        <p>
         Mission accomplished! It’s one (wrapped) line of code. However, this led me to wonder if I could parallelize this. So, my next test was to make a parallel stream of numbers that I then filtered to those that cleanly divided the number being tested. I then summed these, checked for equality to the original number, and returned that.
        </p>
        <pre class="brush: java; title: ; notranslate" title="">
 public static boolean parallelFilteredIsPerfect(long num) {
   return LongStream.rangeClosed(1, num / 2)
     // make this a parallel stream so we can find the whole divisors more quickly (ideally)
     .parallel()
     // filter out any non-whole divisors
     .filter(test -&gt; num % test == 0)
     .sum() == num;
 }
 </pre>
        <p>
         This too worked, though arguably it’s more than one line of code.
        </p>
        <p>
         This success made me wonder how much (if at all) more efficient this method was than my first attempt. I wrote tests for the first six perfect numbers and found that the sixth one, 8,589,869,056, took quite a while to process. My first attempt with
         <code>
          reduceIsPerfect()
         </code>
         took about 45 secs to validate. My second attempt,
         <code>
          parallelFilteredIsPerfect()
         </code>
         took 23 seconds, so it was about two times faster. But, this left a bad taste in my mouth. 23 seconds is a
         <em>
          really
         </em>
         long time.
        </p>
        <p>
         I tried a few other approaches, but found that, in general, most of my attempts tested in the 45 to 50 second range, which was disappointing.
        </p>
        <p>
         So, next, I did what any good programmer would do, I turned to Google. I reasoned that the part of the process that takes the most time is determining if a number is a divisor or not. Google led me to a more efficient algorithm.
        </p>
        <p>
         The way I was testing for perfection was by looping from 1 to half the number being tested. At each step I tested to see if the number could be divided cleanly. For example, with 28 I was doing this:
        </p>
        <p>
         Start with a
         <code>
          sum
         </code>
         of 0. Loop from 1 to 14 (inclusive).
        </p>
        <ol>
         <li>
          <code>
           28 % 1
          </code>
          . Yes! Add 1 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 1.
         </li>
         <li>
          <code>
           28 % 2
          </code>
          . Yes! Add 2 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 3.
         </li>
         <li>
          <code>
           28 % 3
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 4
          </code>
          . Yes! Add 4 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 7.
         </li>
         <li>
          <code>
           28 % 5
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 6
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 7
          </code>
          . Yes! Add 7 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 14.
         </li>
         <li>
          <code>
           28 % 8
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 9
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 10
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 11
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 12
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 13
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 14
          </code>
          . Yes! Add 14 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 28.
         </li>
        </ol>
        <p>
         So, 28 is perfect.
        </p>
        <p>
         But, Google pointed out to me that half of this information is already known by the time I calculate it. The divisors are 1, 2, 4, 7, 14, 28. As soon as I discover 1 I can quickly determine that it’s paired with 28. 2 is paired with 14, and 4 with 7. I don’t need to continue working at all once I’ve gone to 4! This is less than the square root of 28, 5.291502622129181.
        </p>
        <p>
         So, if I loop from 1 to the square root of the number being tested, I can just take the number being tested as well as it’s paired divisor
         <em>
          at the same time
         </em>
         . Add these to the sum and I’m in business:
        </p>
        <p>
         Start with a
         <code>
          sum
         </code>
         of 1. Loop from 2 to the square root of 28 (rounded down to 5) (inclusive).
        </p>
        <ol start="2">
         <li>
          <code>
           28 % 2
          </code>
          . Yes! Add 2 and 14 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 17.
         </li>
         <li>
          <code>
           28 % 3
          </code>
          . No!
         </li>
         <li>
          <code>
           28 % 4
          </code>
          . Yes! Add 4 and 7 to
          <code>
           sum
          </code>
          .
          <code>
           sum
          </code>
          is now 28.
         </li>
         <li>
          <code>
           28 % 5
          </code>
          . No!
         </li>
        </ol>
        <p>
         And, again, I’ve confirmed that 28 is perfect.
        </p>
        <p>
         With this in mind, I rewrote my original one line solution like this:
        </p>
        <pre class="brush: java; title: ; notranslate" title="">
 public static boolean reduceSqrtIsPerfect(long num) {
   return LongStream.rangeClosed(2, (long) Math.sqrt(num))
     .reduce(1, (sum, test) -&gt; num % test == 0 ? sum + test + (num / test) : sum) == num;
 }
 </pre>
        <p>
         And viola, my test of 8,589,869,056 went from taking 23 seconds to
         <em>
          3 milliseconds
         </em>
         !! After all, rather than looping from 1 to 4294934528 and testing each step along the way, I am now looping from 2 to 92681! This is
         <em>
          significantly
         </em>
         less work.
        </p>
        <p>
         And so, I’m fairly happy with the result. That, and, after all that, it’s just one line of code.
        </p>
        <p>
         The
         <a href="https://github.com/dhughes/PerfectNumbers">
          full code and tests for this project have been posted to Github
         </a>
         here.
        </p>

        <h2>

        <p>

         <em>

         </em>

        </p>

 int total = scores.stream().mapToInt(Scoreable::getScore).sum();
 return total / scores.size();
 </pre>
        <p>
         This Java code could be translated into english as something like, “Calculate the average score from a set of scores.” This example is intended to illustrate a point:
         <em>
          Java isn’t english
         </em>
