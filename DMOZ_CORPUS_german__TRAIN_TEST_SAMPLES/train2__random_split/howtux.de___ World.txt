      <h1 style="display: inline;">
       <a class="headline" href="index.php?article=123">
        Virtuelle Maschinen mit KVM
       </a>
      </h1>
     </div>
     <div class="subheadline" style="position: relative; width: 100%; z-index: 0; height: 3em;">
      <span style="float: left;">
       Kategorie: Client / Server Dienste
      </span>
      <span style="float: right; right: 0px; text-align: right;">
       Erstellt: 03.05.2010 - 21:56 Geändert: 26.06.2013 - 13:46
       <br/>
       Autor: Stefan Boeke
      </span>
     </div>
     <div class="article_text" style="position: relative; z-index: 0;">
      Um auf einem Server virtuelle Maschinen mit KVM zu realisieren sind einige Schritte notwendig. In diesem Beispiel wird davon ausgegangen, dass wir einen Server mit einem Netzwerkinterface und mehreren IP-Adressen betreiben wollen. Jede IP-Adresse soll einer virtuellen Maschine (hier ein Ubuntu Server 10.4 LTS) zugeordnet werden. Der Host erhält in diesem Beispiel die IP-Adresse 47.11.8.15 . Die Gäste erhalten die IP-Adressen 47.11.8.17 bis 47.11.8.20. Für ein für die hier gezeigte Konfiguration wird eine Gateway-IP-Adresse benötigt, diese lautet hier 47.11.8.16.
      <br/>
      <br/>
      <h3 class="article">
       Vorraussetzungen
      </h3>
      <br/>
      Für den Betrieb von KVM werden die folgenden Pakete kvm und qemu benötigt. Das Installieren dieser Pakete erfolgt mittels des folgenden Befehls:
      <br/>
      <pre class="code">Installation benötigter Pakete:
 <br/>root@host~ # apt-get install qemu-kvm kvm </pre>
      <!--code-->
      <br/>
      Des Weiteren müssen die Hardwarevirtualisierungen der im System befindlichen CPU aktiviert werden. Dies kann man herausfinden, indem man sich die Datei /proc/cpuinfo anschaut und nach Flags für Hardwarevirtualisierung durchsucht. Am besten geht dies mit dem folgenden Befehl:
      <br/>
      <pre class="code">Unterstützt die CPU Hardwarevirtualisierung?
 <br/>root@host~ # grep -E '^flags.*b(vmx|svm)b' /proc/cpuinfo </pre>
      <!--code-->
      <br/>
      Wenn als Reaktion auf diesen Befehl unter
      <i>
       flags
      </i>
      entweder der Eintrag
      <i>
       vmx
      </i>
      oder
      <i>
       svm
      </i>
      vorhanden ist, beherrscht der Prozessor Hardwarevirtualisierung und KVM kann genutzt werden. Ist dies nicht der Fall können Sie KVM mit diesem Prozessortyp nicht nutzen.
      <br/>
      <p class="opt">
       Hardwarevirtualisierung ist für den Einsatz von KVM notwendig
       <br/>
       Um KVM nutzen zu können muss der Prozessor des Hostsystems Hardwarevirtualisierung unterstützen. Häufig ist die Hardwarevirtualisierung der CPU im Bios des Rechners deaktiviert. Sollten Sie sich sicher sein, dass Ihre CPU Hardwarevirtualisierung unterstützt sollten Sie die Einstellung im Bios ihres Rechners überprüfen, falls der eben beschriebene Test fehlschlägt.
      </p>
      <!--opt-->
      <br/>
      Unterstützt der Prozessor Hardwarevirtualisierung, so werden in der Regel die beiden benötigten Kernel-Module beim Start des Systems automatisch geladen. Dies kann folgendermaßen geprüft werden:
      <br/>
      <pre class="code">Prüfen ob benötigte Kernelmodule geladen wurden:
 <br/>root@host~ # lsmod | grep kvm 
 <br/>Die Ausgabe sollte so aussehen:
 <br/><b>## für Intel-Prozessoren</b>
 <br/>kvm
 <br/>kvm_intel
 <br/><b>## für AMD-Prozessoren</b>
 <br/>kvm
 <br/>kvm_amd</pre>
      <!--code-->
      <br/>
      Sollten die Ausgaben nicht wie beschrieben erscheinen müssen die Module von Hand nachgeladen werden. Dies geschieht folgendermaßen mit dem Befehl modprobe:
      <br/>
      <pre class="code">Manuelles laden der benötigten Kernelmodule:
 <br/><b>## für Intel-Prozessoren</b>
 <br/>root@host~ # modprobe kvm 
 <br/>root@host~ # modprobe kvm_intel
 <br/><b>## für AMD-Prozessoren</b>
 <br/>root@host~ # modprobe kvm 
 <br/>root@host~ # modprobe kvm_amd</pre>
      <!--code-->
      <br/>
      <br/>
      <h3 class="article">
       Erstellen eines KVM-Images
      </h3>
      <br/>
      Zunächst muss eine virtuelle Festplatte für die Aufnahme des virtuellen Gastes angelegt werden:
      <br/>
      <pre class="code">Anlegen einer virtuellen Festplatte für einen KVM-Gast:
 <br/>root@host~ # qemu-img create lucid-lynx-64.img 30G</pre>
      <!--code-->
      <br/>
      Dieser Befehl legt ein 30Gb großes (noch leeres) Image im aktuellen Ordner ab. Das Image erhält hier den Namen lucid-lynx-64.img. Nun kann dieses Image mit Leben gefüllt werden. Hierfür wird ein aktuellen CD-Image von Ubuntu-Server 10.4 heruntergeladen.
      <br/>
      <pre class="code">Herunterladen des Installations-Images (hier 64Bit-Version):
 <br/>root@host~ # wget <a href="http://releases.ubuntu.com/releases/10.04/ubuntu-10.04.1-server-amd64.iso" target="_blank">http://releases.ubuntu.com/releases/10.04/ubuntu-10.04.1-server-amd64.iso</a> </pre>
      <!--code-->
      <br/>
      Nachdem der Download abgeschlossen wurde kann mit der Installation des Betriebssystems in den das KVM-Image begonnen werden.
      <br/>
      <pre class="code">Installieren des Betriebssystems in der virtuellen Festplatte des Gastes:
 <br/><b># Bei 64Bit-Gastsystemen:</b>
 <br/>root@host~ # qemu-system-x86_64 -hda lucid-lynx-64.img -cdrom ubuntu-10.04-server-amd64.iso -boot d -m 1024 -vnc :1
 <br/><b># Bei 32Bit-Gastsystemen:</b>
 <br/>root@host~ # qemu -hda lucid-lynx-64.img -cdrom ubuntu-10.04-server-amd64.iso -boot d -m 1024 -vnc :1</pre>
      <!--code-->
      <p class="opt">
       Verwendete Optionen:
       <br/>
       <b>
        -hda [Dateiname]
       </b>
       <br/>
       Angabe des zuvor angelegten Festplattenimages. Dieses wird als Ziel für die Installation genutzt.
       <br/>
       <b>
        -cdrom [Dateiname|Laufwerk]
       </b>
       <br/>
       Angabe der zu Bootenden Quelle für das zu installierende Betriebssystem. Entweder wird hier ein Image der Installations-CD oder ein Device mit der eingelegten CD (z.B. /dev/cdrom) angegeben.
       <br/>
       <b>
        -boot [a|c|d]
       </b>
       <br/>
       Angabe des Mediums von dem gestartet werden soll. 'a' weisst qemu an von einer Diskette zu booten, 'c' lässt qemu von dem unter -hda angegebenen Image starten und 'd' veranlasst qemu von der unter -cdrom angebenen Quelle zu booten (dies ist hier erforderlich)
       <br/>
       <b>
        -m [Speichergröße]
       </b>
       <br/>
       Angabe des Hauptspeichers der der virtuellen Maschine zur Verfügung gestellt werden soll (z.B. 1024M). Wichtig: Die hier angegebene Speichergröße darf die Größe des physisch auf dem Hostsystems zur Verfügung stehenden Speichers nicht übersteigen.
       <br/>
       <b>
        -vnc [Display]
       </b>
       <br/>
       Gibt an mit welchen VNC-Display die virtuelle Maschine verbunden werden soll. Die hier angegebene Zahl + 5900 ergibt den Port, welcher bei der Verbindung mit einem VNC-Client-Programm angegeben werden muss (in diesem Beispiel entspricht dies Port 5901)
       <br/>
      </p>
      <!--opt-->
      <br/>
      Der Installationsprozess kann nun mittels eines VNC-Clients erfolgen. Unter Kubuntu kann beispielsweise das Programm KDRC verwendet werden. Als Kommunikationspartner für KDRC wird die IP-Adresse des Hosts angegeben (47.11.8.15). Da wir die Installation mit dem Parameter -vnc 1 gestartet haben muss zusätzlich der Port 5901 eingegeben werden. Sobald man sich nun mit dem Server verbunden hat kann der Installationsprozess wie gewohnt durchgeführt werden.
      <br/>
      <br/>
      <h3 class="article">
       Allgemeine Konfiguration des Netzwerkes auf dem Host
      </h3>
      <br/>
      Um Netzwerkverkehr über den Host zu leiten muss auf diesem IP-Forwarding aktiviert werden. Dies geschieht, indem man in die Datei ip_forward eine einfache Eins einfügt.
      <br/>
      <pre class="code">Freischalten von IP-Forwarding auf dem Host:
 <br/>root@host~ # echo 1 &gt; /proc/sys/net/ipv4/ip_forward
 <br/></pre>
      <!--code-->
      <br/>
      Danach legen wir eine Netzwerkbridge an. Diese Bridge dient später allen virtuellen Maschinen als Gateway und erhält die hierfür reservierte IP-Adresse
      <br/>
      <pre class="code">Anlegen einer Netzwerkbridge:
 <br/>root@host~ # brctl addbr br01
 <br/># Zuweisen der Gateway-Adresse:
 <br/>root@host~ # ifconfig br01 47.11.8.16 netmask 255.255.255.0 up</pre>
      <!--code-->
      <br/>
      <p class="opt">
       Alle bis hierhin gemachten Netzwerkkonfigurationen sind nur genau einmal zu machen. Alle KVM-Gäste nutzen diese Konfigurationen gemeinsam.
      </p>
      <!--opt-->
      <br/>
      <br/>
      <h3 class="article">
       Individuelle Konfiguration des Netzwerkes für jeden Gast
      </h3>
      <br/>
      Für jeden KVM-Gast wird zunächst ein TAP-Device angelegt. In diesem Beispiel erhält der erste KVM-Gast das TAP-Device qtap0. Der nächste Gast würde demnach qtap1, der übernächste qtap2 u.s.w. bekommen.
      <br/>
      <pre class="code">Anlegen eines tap-Device:
 <br/>root@host~ # tunctl -b -u root -t qtap0</pre>
      <!--code-->
      <br/>
      Nachdem dies erledigt wurde wird nun eine Verknüpfung zwischen der bereits angelegten Netzwerkbridge und dem soeben angelegten TAP-Device hergestellt.
      <br/>
      <pre class="code">Zuweisen des Tap-Device der Bridge:
 <br/>root@host~ # brctl addif br01 qtap0</pre>
      <!--code-->
      <br/>
      Danach kann das TAP-Device hochgefahren (aktiviert) werden. Damit alle Netzwerkpakete wirklich den Gast erreichen wird das TAP-Device im Promiscous-Mode und ohne IP-Adresse betrieben.
      <br/>
      <pre class="code">Hochfahren des Tap-Device:
 <br/>root@host~ # ifconfig qtap0 up 0.0.0.0 promisc</pre>
      <!--code-->
      <br/>
      Durch das Anlegen einer Host-Route wird der an die IP-Adresse des Gastes gerichtete Netzwerkverkehr an die Netzwerkbridge geleitet und erreicht dort über qtap0 die virtuelle Maschine. Hier ist es wichtig die korrekte IP-Adresse des Gastes anzugeben.
      <br/>
      <pre class="code">Anlegen einer Route für den Gast:
 <br/>root@host~ # route add -host 47.11.8.17 dev br01</pre>
      <!--code-->
      <br/>
      <br/>
      <h3 class="article">
       Starten der virtuellen Maschine
      </h3>
      <br/>
      Sobald die Vorbereitungen bis hierhin abgeschlossen wurden kann die virtuelle Maschine gestartet werden. Die meisten Optionen sind bereits weiter oben in diesem Artikel beschrieben worden, zu allen anderen finden Sie die Erläuterungen weiter unterhalb des folgenden Beispiels.
      <br/>
      <pre class="code">Starten des Gastsystems:
 <br/><b>Für 64Bit-Gastsysteme:</b>
 <br/>root@host~ # qemu-system-x86_64 -k de -m 1024 -smp 1 -hda lucid-lynx-64.img 
 <br/>         -net nic,model=rtl8139,macaddr=00:92:aa:44:23:01 -net 
 <br/>          tap,ifname=qtap0,script=no -vnc :1
 <br/><b>Für 32Bit-Gastsysteme:</b>
 <br/>root@host~ # qemu -k de -m 1024 -smp 1 -hda lucid-lynx-64.img 
 <br/>         -net nic,model=rtl8139,macaddr=00:92:aa:44:23:01 -net 
 <br/>          tap,ifname=qtap0,script=no -vnc :1</pre>
      <!--code-->
      <br/>
      <p class="opt">
       Optionen:
       <br/>
       Comming soon...
       <br/>
      </p>
      <!--opt-->
      <br/>
      Nachdem der Gast gestartet wurde kann nun mittels der VNC-Client-Software eine Verbindung aufgebaut werden. Nach dem Login sind evtl. die Netzwerkeinstellungen anzupassen (IP-Settings, DNS, etc.). Danach sollte der Gast erreichbar sein und kann genutzt werden.
      <br/>
      <p class="opt">
       Um Netzwerk innerhalb der virtuellen Maschine nutzen zu können muss eventuell noch das passende Modul geladen werden. Dies geschieht mit:
       <br/>
       <i>
        modprobe 8139too
       </i>
      </p>
      <!--opt-->

       </a>
      </h1>
     </div>
     <div class="subheadline" style="position: relative; width: 100%; z-index: 0; height: 3em;">
      <span style="float: left;">
       Kategorie: Security
      </span>
      <span style="float: right; right: 0px; text-align: right;">
       Erstellt: 19.10.2008 - 17:26 Geändert: 19.10.2008 - 17:46
       <br/>
       Autor: Stephan Küffner
      </span>
     </div>
     <div class="article_text" style="position: relative; z-index: 0;">
      <b>
       Erstellen eines X.509 Server Zertifikat
      </b>
      <br/>
      <br/>
      Zertifikate sind wir digitale Unterschriften. Unter Debian-Linux wird ein Zertifikat wird mit dem Programm openssl erstellt.
      <br/>
      Die Installation auf dem Server geschieht mit
      <b>
       apt-get install openssl
      </b>
      .
      <br/>
      <br/>
      <pre class="code">Zertifikatserstellung auf dem Server
 <br/>sv1@server:~$ openssl req -new -x509 -nodes -out SERVERNAME.crt -keyout SERVERNAME.key 
 <br/>Generating a 1024 bit RSA private key
 <br/>........................................................................++++++
 <br/>...++++++
 <br/>writing new private key to 'SERVERNAME.key'
 <br/>
 <br/>You are about to be asked to enter information that will be incorporated
 <br/>into your certificate request.
 <br/>What you are about to enter is what is called a Distinguished Name or a DN.
 <br/>There are quite a few fields but you can leave some blank
 <br/>For some fields there will be a default value,
 <br/>If you enter '.', the field will be left blank.
 <br/>
 <br/>Country Name (2 letter code) [AU]:EU
 <br/>State or Province Name (full name) [Some-State]:STATE
 <br/>Locality Name (eg, city) []:CITY
 <br/>Organization Name (eg, company) [Internet Widgits Pty Ltd]:ORG
 <br/>Organizational Unit Name (eg, section) []: NAMEOFORG
 <br/>Common Name (eg, YOUR name) []:SERVERNAME
 <br/>Email Address []: WEBMASTER
 <br/></pre>
      <!--code-->
      <br/>
      <br/>
      <p class="def">
       Dieses Zertifikat ist selbstsigniert. Jeder gängige Browser wird dieses Zertifikat anfänglich als nicht vertrauenswürdig einstufen, da er nicht die Zertifikatshierarchie bis zur RootCA auflösen kann.
      </p>
      <!--def-->
      <br/>
      <p class="opt">
       Offizielle und voll vertrauenswürdige Zertifikate stellen Firmen wie CaCert.org oder VeriSign.de aus
      </p>
      <!--opt-->
      <br/>
      <br/>
      Beim Erstellen des Zertifikats kann man noch unterschiedliche Attribute angeben. Ein sinnvolles noch zu erwähnendes Attribut ist:
      <br/>
      <pre class="code">Einzelnes Attribut:
 <br/>-days 365
 <br/>Vollständiger Befehl könnte dann folgendermaßen lauten:
 <br/>sv1@server:~$ openssl req -new -x509 -nodes -days 365 -out SERVERNAME.crt -keyout
 <br/></pre>
      <!--code-->
      <br/>
      Damit erreicht man, dass das selbstsignierte Zertifikat 1 Jahr Gültigkeit besitzt.
      <br/>
      <br/>
      Für mehr Informationen zu den Attributen unter:
      <br/>
      <pre class="code">sv1@server:~$ man openssl</pre>
      <!--code-->
